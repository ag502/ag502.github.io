만약 중간이나 큰 규모의 리액트 애플리케이션을 만들어 본 적이 있다면, 컴포넌트들의 리렌더가 많이 발생한다는 것을 느꼈을 것입니다.
어떤 경우에는 이것이 성능 이슈로 이어질 수 있습니다. 애플리케이션 구조에서 상위에 위치하고 있는 컴포넌트의 작은 변경이 하위의 연관 없는 다수의 컴포넌트들의 재렌더링을 야기할 수 있습니다.

이 글에서는 컴포넌트의 재렌더링이 왜 발생하는지 살펴볼 것입니다. 또한 이와 관련하여 최적화할 수 있는 세 가지 리액트 도구를 살펴보겠습니다.

- `React.memo`
- `useCallback`
- `useMemo`

## 🌃 리액트에서 재렌더링이 발생하는 이유

`useMemo`와 `useCallback`을 살펴보기 전에, 리액트의 렌더링 과정에 익숙해져야 합니다.

앞에서 상태 설정 함수(`setCount`, `setUser`)가 재렌더링을 어떻게 트리거하는지를 살펴봤습니다.
리액트는 새로운 상태값으로 UI가 어떻게 표현되어야 하는지를 스냅샷으로 찍습니다.

사실, 우리는 재렌더링에 대하여 많은 것을 살펴봤습니다. 이 섹션에서는 살펴본 내용들을 다듬고 더 명확하게 이해할 수 있도록 해보겠습니다.

### 🎆 기본적인 규칙

가장 근본적인 내용부터 시작해보겠습니다. 모든 재렌더링은 상태 변경에서 시작됩니다. 상태 변경은 리액트에서 컴포넌트의 재렌더링을 트리거하는 유일한 요소입니다.

아마 조금 이상하게 들릴 수도 있습니다. 컴포넌트는 props가 변경될 때 다시 렌더링되는 게 아닌가요?

하지만 사실은 그렇지 않습니다. 컴포넌트가 다시 렌더링될 때 그 컴포넌트의 모든 하위 컴포넌트들도 다시 렌더링됩니다.

```jsx
function App() {
  return (
    <>
      <Counter />
      <footer>
        <p>Copyright 2022 Big Count Inc.</p>
      </footer>
    </>
  );
}

function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <main>
      <BigCountNumber count={count} />
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </main>
  );
}

function BigCountNumber({ count }) {
  return (
    <p>
      <span className="prefix">Count:</span>
      {count}
    </p>
  );
}
```

위 예시에는 세 개의 컴포넌트가 있습니다. 상위에 `App` 컴포넌트가 있고 이 컴포넌트는 `Counter` 컴포넌트를 렌더링합니다.
`Counter` 컴포넌트는 또 `BigCountNumber`를 렌더링합니다.

리액트에서 모든 상태 변수는 해당하는 컴포넌트의 인스턴스에 포함됩니다.
위 코드에서 `count`라는 단일 상태가 있으며, 이 상태는 `Counter` 컴포넌트와 연관이 있습니다.

상태가 변경될 때마다, `Counter`는 다시 렌더링됩니다. `BigCounterNumber`가 `Counter`에 의해 렌더링되기 때문에, `BigCounterNumber` 또한 다시 렌더링됩니다.

좋습니다. 이제 첫 번째 오개념부터 바로잡아 봅시다. 상태 변수가 변경될 때마다 앱 전체가 다시 렌더링된다는 생각은 잘못된 것입니다.

재렌더링은 변경된 상태를 소유한 컴포넌트와 그의 자식 컴포넌트에서만 일어납니다.
위 예시에서 `App` 컴포넌트는 `count` 상태 변수가 변경되어도 다시 렌더링되지 않습니다.

이 규칙을 암기하기보다 한 걸음 물러서서 왜 이렇게 동작하는지를 살펴보겠습니다.

리액트의 "주된 역할"은 애플리케이션 UI를 리액트의 상태와 동기화하는 것입니다. 재렌더링의 핵심은 무엇을 바꿔야 하는지 파악하는 것입니다.

위의 "Counter" 예시를 살펴보겠습니다. `Counter` 컴포넌트가 최초로 마운트되었을 때, 리액트는 모든 연관된 요소를 렌더링하며 DOM이 어떻게 생겨야 하는지에 대하여 아래의 스케치를 그립니다.

```html
<main>
  <p>
    <span class="prefix">0</span>
  </p>
  <button>Increment</button>
</main>
```

사용자가 버튼을 클릭하면 `count` 상태는 `0`에서 `1`로 변경됩니다. 어떻게 이것이 UI에 영향을 미칠까요?
그 점이 재렌더링 작업을 통해 알아볼 내용입니다.

리액트는 `Counter`와 `BigCounter` 컴포넌트의 코드를 재실행하여 새로운 DOM의 모습을 생성합니다.

```html
<main>
  <p>
    <span class="prefix">Count:</span>
    1
  </p>
  <button>Increment</button>
</main>
```

앞서 살펴보았듯이, 매 렌더링은 스냅샷과 같습니다. 이 스냅샷은 현재의 상태에 기반하여 UI가 어떤 모습인지를 알려주는 사진과도 같습니다.

스냅샷을 생성한 다음, 리액트는 스냅샷 사이의 다른 점을 찾습니다. 이 경우 paragraph의 텍스트 노드가 `0`에서 `1`로 변경되었고 이를 반영하기 위해 DOM의 텍스트 노드를 변경해야 합니다.
이 과정들이 다 마무리되면, 리액트는 다음 상태 변화를 기다리며 휴지 상태에 들어갑니다.

`count` 상태는 `Counter` 컴포넌트와 연관이 있습니다. 리액트 애플리케이션에서 데이터는 위로 올라갈 수가 없기 때문에 상태의 변경은 `<App/>`에 영향을 끼칠 수 없습니다. 따라서 해당 컴포넌트는 다시 렌더링되지 않습니다.

하지만 `Counter`의 자식인 `BigCountNumber`는 다시 렌더링되어야 합니다. 이 컴포넌트는 실제로 `count` 상태를 보여주는 컴포넌트입니다. 만약에 렌더링되지 않는다면, paragraph의 텍스트 노드가 `0`에서 `1`로 바뀌는 것을 알 수 없습니다.
따라서 이 컴포넌트를 스케치에 추가해야 합니다.

재렌더링의 핵심은 상태 변경이 UI에 어떻게 영향을 끼치는지 알아내는 것입니다. 그리고 정확한 스냅샷을 만들기 위해 연관 있는 모든 컴포넌트를 다시 렌더링해야 합니다.

### 🎆 Props는 중요하지 않습니다.

좋습니다. 이제 두 번째 오개념을 살펴보겠습니다. 컴포넌트는 props가 변경될 때 다시 렌더링되지 않습니다.

앞에서 살펴본 코드를 변경해보겠습니다.

"Counter" 앱에서 새로운 `Decoration` 컴포넌트를 추가했습니다.

```jsx
function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <main>
      <BigCountNumber count={count} />
      <button onClick={() => setCount(count + 1)}>Increment</button>

      <Decoration />
    </main>
  );
}
```

`Decoration` 컴포넌트는 counter 앱에 귀여운 이모지를 추가해주는 역할을 합니다. 이 컴포넌트는 `count`에 의존하지 않습니다.
따라서 `count`가 변경되어도 이 컴포넌트는 다시 렌더링되지 않는 게 맞겠죠?

하지만 그렇지 않습니다.

<img src={PropsRerender} width="100%" height="auto" />

컴포넌트가 재렌더링될 때, 모든 자식 컴포넌트들을 다시 렌더링합니다. 자식 컴포넌트들이 업데이트되는 상태를 props로 전달받은 것과는 무관합니다.

사실 납득하기가 조금 어렵습니다. `count`를 props로 받지 않은 `Decoration` 컴포넌트는 왜 다시 렌더링되는 걸까요?

이 질문에 대한 답은, 리액트는 `Decoration` 컴포넌트가 직접적으로든 간접적으로든 `count` 상태에 의존하는지 확인할 수 없기 때문입니다.

이상적인 세계에서 리액트 컴포넌트는 항상 "순수"해야 합니다. 순수한 컴포넌트는 같은 props를 받았을 때 항상 같은 UI를 생성합니다.

하지만 현실 세계에서 대부분의 컴포넌트는 순수하지 않습니다. 순수하지 않은 컴포넌트를 만드는 방법은 놀라울 정도로 쉽습니다.

```jsx
function CurrentTime() {
  const now = new Date();

  return <p>The time is {now.toString()}.</p>;
}
```

위 컴포넌트는 현재 시간을 계산하기 때문에 렌더링될 때마다 다른 모습으로 렌더링됩니다.

이보다 더 알아차리기 어려운 경우는 ref와 관련이 있습니다. ref를 props로 전달하면, 리액트는 마지막 렌더링 이후 ref가 변경되었는지 여부를 알 수 없습니다. 따라서 안전을 위해 재렌더링을 선택합니다.

리액트의 최우선 목표는 UI를 상태와 동기화하는 것입니다. 따라서 렌더링 횟수를 늘리는 쪽으로 방향을 설정했습니다. 사용자에게 오래된 UI를 보여주는 위험을 감수하고 싶지 않기 때문입니다.

정리해보자면 _props는 재렌더링과는 무관합니다._ `BigCounterNumber` 컴포넌트는 `count` props가 변경되어 다시 렌더링된 것이 아닙니다.

상태 변수가 변경되어 해당하는 컴포넌트가 다시 렌더링될 때, 새로운 스냅샷을 구성하기 위해 트리를 아래로 순회하면서 연쇄적으로 재렌더링을 수행합니다.

하지만 이 과정을 최적화하기 위해 우리가 할 수 있는 일이 있습니다. 이는 다음 섹션에서 살펴보도록 하겠습니다.

:::info Rendering vs. Painting
[렌더링이 페인팅과 다르다](/blog/useState-hook#-rendering-vs-painting)는 점을 아는 것이 중요합니다. 재렌더링은 변경사항이 없으면 DOM을 수정하지 않습니다.

불필요한 재렌더링은 성능 저하의 원인이 될 수 있으며, 특히 reconciliation 과정에서 많은 요소를 비교해야 한다면 더욱 그렇습니다.
뒤에서 이 문제를 해결할 수 있는 방법을 알아보겠습니다만, 대부분의 경우 재렌더링은 무시할 수 있는 수준이라는 것을 기억하는 것이 중요합니다.
리액트는 충분히 효율적이고 빠르게 설계되었기 때문에 세세하게 최적화할 필요가 없습니다.
:::

## 🌃 순수한 컴포넌트

앞의 섹션에서 props의 변경과는 상관없이 부모 컴포넌트가 재렌더링될 때 자식 컴포넌트가 자동으로 재렌더링됨을 살펴봤습니다.

큰 규모의 애플리케이션에서는 하나의 상태가 변경되면 실제로 다시 렌더링이 필요한 컴포넌트는 소수임에도 불구하고 수십, 수백 개의 컴포넌트가 재렌더링되어 성능 이슈로 이어질 수 있습니다.

다행이도, 리액트는 `React.memo`라는 도구를 제공하여 이를 해결할 수 있게 합니다.

```jsx
function Decoration() {
  return <div className="decoration">⛵</div>;
}

const PureDecoration = React.memo(Decoration);

export default PureDecoration;
```

`React.memo`는 리액트에게 해당 컴포넌트가 순수한 컴포넌트임을 알려주는 역할을 합니다. 따라서 컴포넌트의 props나 상태가 변경되지 않는 한 다시 렌더링할 필요가 없습니다. 이 컴포넌트는 동일한 props나 state가 주어지면 항상 동일한 UI를 반환합니다.

`React.memo`는 인자로 컴포넌트(`Decoration`)를 받아, 해당 컴포넌트와 관련 없는 재렌더링을 선택적으로 무시하는 기능을 추가해 줍니다.

부모 컴포넌트가 다시 렌더링될 때, 리액트는 자식 컴포넌트인 `PureDecoration`를 다시 렌더링하려고 하지만 해당 컴포넌트의 props는 변경되지 않았으므로 재렌더링을 하지 않습니다.

이런 기술을 _memoization_ 이라고 합니다.

Memoization을 할 때 리액트는 이전의 스냅샷을 기억해둡니다. 만약 props가 변경되지 않는다면 리액트는 새로운 스냅샷을 생성하지 않고 이전의 스냅샷을 재사용합니다.

`React.memo`를 `BigCountNumber`와 `Decoration` 컴포넌트에 적용해보겠습니다.

<img src={ReactMemo} width="100%" height="auto" />

`count`가 변경되면, `Counter` 컴포넌트를 재렌더링 합니다. 이 때 리액트는 두 개의 자식 컴포넌트도 다시 렌더링하려고 합니다.

`BigCountNumber` 컴포넌트가 `count`를 props로 받고 있고, `props`가 변경되었기 때문에 `BigCountNumber` 컴포넌트는 다시 렌더링됩니다.
하지만 `Decoration` 컴포넌트의 props는 변경되지 않았기 때문에 기존의 스냅샷이 재사용됩니다.

저는 `React.memo`를 게으른 사진사로 비유하고 싶습니다. 만약 똑같은 사물을 5장 찍어달라고 요청하면 사진사를 1장만 찍고 나머지는 복사해 줄 것입니다.
사진사는 요청사항이 바뀔 때만 새로운 사진을 찍습니다.

정리하자면

- 리액트에서 재렌더링을 트리거할 수 있는 유일한 방법은 상태 설정 함수를 사용해 상태를 업데이트하는 것입니다.
- 컴포넌트가 재렌더링될 때, props가 변경되지 않아도 모든 자식 컴포넌트를 다시 렌더링합니다.
- `React.memo`를 컴포넌트에 적용하여 최적화할 수 있습니다. 이렇게 하면 props 중에 하나라도 변경될 때 다시 렌더링합니다.

:::info 왜 memoization이 기본 동작이 아닌가요?
`React.memo`의 동작이 기본 동작이 아닌지 궁금할 것입니다.
다시 렌더링을 할 필요가 없는 컴포넌트의 렌더링을 건너뛰면 성능을 확실히 향상시킬 수 있습니다.

개발자로서 재렌더링에 드는 비용을 과대평가하는 경향이 있습니다. `Decoration` 컴포넌트의 재렌더링시에 드는 시간은 정말 빠릅니다.
컴포넌트에 props는 많지만 자식 컴포넌트가 많지 않은 경우에는, props 변경 여부를 확인하는 비용이 재렌더링보다 더 들 수도 있습니다.

따라서 공식적인 추천은 성능적인 이슈를 해결하기 위해 `React.memo`를 필요할 때 사용하는 것입니다. 섣부른 최적화를 하지 않도록 주의해야 합니다.
:::

<summary>섹션 요약</summary>

- `React.memo`는 컴포넌트를 감싸서 props가 변경되지 않으면 재렌더링을 건너뛰는 memoization 기법이다.
- props가 변경된 컴포넌트만 다시 렌더링되고, 변경되지 않은 컴포넌트는 이전 스냅샷을 재사용한다.
- memoization이 기본 동작이 아닌 이유는 props 비교 비용이 재렌더링보다 클 수 있기 때문이며, 성능 이슈가 있을 때만 사용하는 것이 권장된다.
