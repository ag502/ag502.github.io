만약 중간이나 큰 규모의 리액트 애플리케이션을 만들어 본 적이 있다면, 컴포넌트들의 리렌더가 많이 발생한다는 것을 느꼈을 것입니다.
어떤 경우에는 이것이 성능 이슈로 이어질 수 있습니다. 애플리케이션 구조에서 상위에 위치하고 있는 컴포넌트의 작은 변경이 하위의 연관 없는 다수의 컴포넌트들의 재렌더링을 야기할 수 있습니다.

이 글에서는 컴포넌트의 재렌더링이 왜 발생하는지 살펴볼 것입니다. 또한 이와 관련하여 최적화할 수 있는 세 가지 리액트 도구를 살펴보겠습니다.

- `React.memo`
- `useCallback`
- `useMemo`

## 🌃 리액트에서 재렌더링이 발생하는 이유

`useMemo`와 `useCallback`을 살펴보기 전에, 리액트의 렌더링 과정에 익숙해져야 합니다.

앞에서 상태 설정 함수(`setCount`, `setUser`)가 재렌더링을 어떻게 트리거하는지를 살펴봤습니다.
리액트는 새로운 상태값으로 UI가 어떻게 표현되어야 하는지를 스냅샷으로 찍습니다.

사실, 우리는 재렌더링에 대하여 많은 것을 살펴봤습니다. 이 섹션에서는 살펴본 내용들을 다듬고 더 명확하게 이해할 수 있도록 해보겠습니다.

### 🎆 기본적인 규칙

가장 근본적인 내용부터 시작해보겠습니다. 모든 재렌더링은 상태 변경에서 시작됩니다. 상태 변경은 리액트에서 컴포넌트의 재렌더링을 트리거하는 유일한 요소입니다.

아마 조금 이상하게 들릴 수도 있습니다. 컴포넌트는 props가 변경될 때 다시 렌더링되는 게 아닌가요?

하지만 사실은 그렇지 않습니다. 컴포넌트가 다시 렌더링될 때 그 컴포넌트의 모든 하위 컴포넌트들도 다시 렌더링됩니다.

```jsx
function App() {
  return (
    <>
      <Counter />
      <footer>
        <p>Copyright 2022 Big Count Inc.</p>
      </footer>
    </>
  );
}

function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <main>
      <BigCountNumber count={count} />
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </main>
  );
}

function BigCountNumber({ count }) {
  return (
    <p>
      <span className="prefix">Count:</span>
      {count}
    </p>
  );
}
```

위 예시에는 세 개의 컴포넌트가 있습니다. 상위에 `App` 컴포넌트가 있고 이 컴포넌트는 `Counter` 컴포넌트를 렌더링합니다.
`Counter` 컴포넌트는 또 `BigCountNumber`를 렌더링합니다.

리액트에서 모든 상태 변수는 해당하는 컴포넌트의 인스턴스에 포함됩니다.
위 코드에서 `count`라는 단일 상태가 있으며, 이 상태는 `Counter` 컴포넌트와 연관이 있습니다.

상태가 변경될 때마다, `Counter`는 다시 렌더링됩니다. `BigCounterNumber`가 `Counter`에 의해 렌더링되기 때문에, `BigCounterNumber` 또한 다시 렌더링됩니다.

좋습니다. 이제 첫 번째 오개념부터 바로잡아 봅시다. 상태 변수가 변경될 때마다 앱 전체가 다시 렌더링된다는 생각은 잘못된 것입니다.

재렌더링은 변경된 상태를 소유한 컴포넌트와 그의 자식 컴포넌트에서만 일어납니다.
위 예시에서 `App` 컴포넌트는 `count` 상태 변수가 변경되어도 다시 렌더링되지 않습니다.

이 규칙을 암기하기보다 한 걸음 물러서서 왜 이렇게 동작하는지를 살펴보겠습니다.

리액트의 "주된 역할"은 애플리케이션 UI를 리액트의 상태와 동기화하는 것입니다. 재렌더링의 핵심은 무엇을 바꿔야 하는지 파악하는 것입니다.

위의 "Counter" 예시를 살펴보겠습니다. `Counter` 컴포넌트가 최초로 마운트되었을 때, 리액트는 모든 연관된 요소를 렌더링하며 DOM이 어떻게 생겨야 하는지에 대하여 아래의 스케치를 그립니다.

```html
<main>
  <p>
    <span class="prefix">0</span>
  </p>
  <button>Increment</button>
</main>
```

사용자가 버튼을 클릭하면 `count` 상태는 `0`에서 `1`로 변경됩니다. 어떻게 이것이 UI에 영향을 미칠까요?
그 점이 재렌더링 작업을 통해 알아볼 내용입니다.

리액트는 `Counter`와 `BigCounter` 컴포넌트의 코드를 재실행하여 새로운 DOM의 모습을 생성합니다.

```html
<main>
  <p>
    <span class="prefix">Count:</span>
    1
  </p>
  <button>Increment</button>
</main>
```

앞서 살펴보았듯이, 매 렌더링은 스냅샷과 같습니다. 이 스냅샷은 현재의 상태에 기반하여 UI가 어떤 모습인지를 알려주는 사진과도 같습니다.

스냅샷을 생성한 다음, 리액트는 스냅샷 사이의 다른 점을 찾습니다. 이 경우 paragraph의 텍스트 노드가 `0`에서 `1`로 변경되었고 이를 반영하기 위해 DOM의 텍스트 노드를 변경해야 합니다.
이 과정들이 다 마무리되면, 리액트는 다음 상태 변화를 기다리며 휴지 상태에 들어갑니다.

`count` 상태는 `Counter` 컴포넌트와 연관이 있습니다. 리액트 애플리케이션에서 데이터는 위로 올라갈 수가 없기 때문에 상태의 변경은 `<App/>`에 영향을 끼칠 수 없습니다. 따라서 해당 컴포넌트는 다시 렌더링되지 않습니다.

하지만 `Counter`의 자식인 `BigCountNumber`는 다시 렌더링되어야 합니다. 이 컴포넌트는 실제로 `count` 상태를 보여주는 컴포넌트입니다. 만약에 렌더링되지 않는다면, paragraph의 텍스트 노드가 `0`에서 `1`로 바뀌는 것을 알 수 없습니다.
따라서 이 컴포넌트를 스케치에 추가해야 합니다.

재렌더링의 핵심은 상태 변경이 UI에 어떻게 영향을 끼치는지 알아내는 것입니다. 그리고 정확한 스냅샷을 만들기 위해 연관 있는 모든 컴포넌트를 다시 렌더링해야 합니다.

### 🎆 Props는 중요하지 않습니다.

좋습니다. 이제 두 번째 오개념을 살펴보겠습니다. 컴포넌트는 props가 변경될 때 다시 렌더링되지 않습니다.

앞에서 살펴본 코드를 변경해보겠습니다.

"Counter" 앱에서 새로운 `Decoration` 컴포넌트를 추가했습니다.

```jsx
function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <main>
      <BigCountNumber count={count} />
      <button onClick={() => setCount(count + 1)}>Increment</button>

      <Decoration />
    </main>
  );
}
```

`Decoration` 컴포넌트는 counter 앱에 귀여운 이모지를 추가해주는 역할을 합니다. 이 컴포넌트는 `count`에 의존하지 않습니다.
따라서 `count`가 변경되어도 이 컴포넌트는 다시 렌더링되지 않는 게 맞겠죠?

하지만 그렇지 않습니다.

<img src={PropsRerender} width="100%" height="auto" />

컴포넌트가 재렌더링될 때, 모든 자식 컴포넌트들을 다시 렌더링합니다. 자식 컴포넌트들이 업데이트되는 상태를 props로 전달받은 것과는 무관합니다.

사실 납득하기가 조금 어렵습니다. `count`를 props로 받지 않은 `Decoration` 컴포넌트는 왜 다시 렌더링되는 걸까요?

이 질문에 대한 답은, 리액트는 `Decoration` 컴포넌트가 직접적으로든 간접적으로든 `count` 상태에 의존하는지 확인할 수 없기 때문입니다.

이상적인 세계에서 리액트 컴포넌트는 항상 "순수"해야 합니다. 순수한 컴포넌트는 같은 props를 받았을 때 항상 같은 UI를 생성합니다.

하지만 현실 세계에서 대부분의 컴포넌트는 순수하지 않습니다. 순수하지 않은 컴포넌트를 만드는 방법은 놀라울 정도로 쉽습니다.

```jsx
function CurrentTime() {
  const now = new Date();

  return <p>The time is {now.toString()}.</p>;
}
```

위 컴포넌트는 현재 시간을 계산하기 때문에 렌더링될 때마다 다른 모습으로 렌더링됩니다.

이보다 더 알아차리기 어려운 경우는 ref와 관련이 있습니다. ref를 props로 전달하면, 리액트는 마지막 렌더링 이후 ref가 변경되었는지 여부를 알 수 없습니다. 따라서 안전을 위해 재렌더링을 선택합니다.

리액트의 최우선 목표는 UI를 상태와 동기화하는 것입니다. 따라서 렌더링 횟수를 늘리는 쪽으로 방향을 설정했습니다. 사용자에게 오래된 UI를 보여주는 위험을 감수하고 싶지 않기 때문입니다.

정리해보자면 _props는 재렌더링과는 무관합니다._ `BigCounterNumber` 컴포넌트는 `count` props가 변경되어 다시 렌더링된 것이 아닙니다.

상태 변수가 변경되어 해당하는 컴포넌트가 다시 렌더링될 때, 새로운 스냅샷을 구성하기 위해 트리를 아래로 순회하면서 연쇄적으로 재렌더링을 수행합니다.

하지만 이 과정을 최적화하기 위해 우리가 할 수 있는 일이 있습니다. 이는 다음 섹션에서 살펴보도록 하겠습니다.

:::info Rendering vs. Painting
[렌더링이 페인팅과 다르다](/blog/useState-hook#-rendering-vs-painting)는 점을 아는 것이 중요합니다. 재렌더링은 변경사항이 없으면 DOM을 수정하지 않습니다.

불필요한 재렌더링은 성능 저하의 원인이 될 수 있으며, 특히 reconciliation 과정에서 많은 요소를 비교해야 한다면 더욱 그렇습니다.
뒤에서 이 문제를 해결할 수 있는 방법을 알아보겠습니다만, 대부분의 경우 재렌더링은 무시할 수 있는 수준이라는 것을 기억하는 것이 중요합니다.
리액트는 충분히 효율적이고 빠르게 설계되었기 때문에 세세하게 최적화할 필요가 없습니다.
:::

## 🌃 순수한 컴포넌트

앞의 섹션에서 props의 변경과는 상관없이 부모 컴포넌트가 재렌더링될 때 자식 컴포넌트가 자동으로 재렌더링됨을 살펴봤습니다.

큰 규모의 애플리케이션에서는 하나의 상태가 변경되면 실제로 다시 렌더링이 필요한 컴포넌트는 소수임에도 불구하고 수십, 수백 개의 컴포넌트가 재렌더링되어 성능 이슈로 이어질 수 있습니다.

다행히도, 리액트는 `React.memo`라는 도구를 제공하여 이를 해결할 수 있게 합니다.

```jsx
function Decoration() {
  return <div className="decoration">⛵</div>;
}

const PureDecoration = React.memo(Decoration);

export default PureDecoration;
```

`React.memo`는 리액트에게 해당 컴포넌트가 순수한 컴포넌트임을 알려주는 역할을 합니다. 따라서 컴포넌트의 props나 상태가 변경되지 않는 한 다시 렌더링할 필요가 없습니다. 이 컴포넌트는 동일한 props나 state가 주어지면 항상 동일한 UI를 반환합니다.

`React.memo`는 인자로 컴포넌트(`Decoration`)를 받아, 해당 컴포넌트와 관련 없는 재렌더링을 선택적으로 무시하는 기능을 추가해 줍니다.

부모 컴포넌트가 다시 렌더링될 때, 리액트는 자식 컴포넌트인 `PureDecoration`를 다시 렌더링하려고 하지만 해당 컴포넌트의 props는 변경되지 않았으므로 재렌더링을 하지 않습니다.

이런 기술을 _memoization_ 이라고 합니다.

Memoization을 할 때 리액트는 이전의 스냅샷을 기억해둡니다. 만약 props가 변경되지 않는다면 리액트는 새로운 스냅샷을 생성하지 않고 이전의 스냅샷을 재사용합니다.

`React.memo`를 `BigCountNumber`와 `Decoration` 컴포넌트에 적용해보겠습니다.

<img src={ReactMemo} width="100%" height="auto" />

`count`가 변경되면, `Counter` 컴포넌트를 재렌더링 합니다. 이 때 리액트는 두 개의 자식 컴포넌트도 다시 렌더링하려고 합니다.

`BigCountNumber` 컴포넌트가 `count`를 props로 받고 있고, `props`가 변경되었기 때문에 `BigCountNumber` 컴포넌트는 다시 렌더링됩니다.
하지만 `Decoration` 컴포넌트의 props는 변경되지 않았기 때문에 기존의 스냅샷이 재사용됩니다.

저는 `React.memo`를 게으른 사진사로 비유하고 싶습니다. 만약 똑같은 사물을 5장 찍어달라고 요청하면 사진사를 1장만 찍고 나머지는 복사해 줄 것입니다.
사진사는 요청사항이 바뀔 때만 새로운 사진을 찍습니다.

정리하자면

- 리액트에서 재렌더링을 트리거할 수 있는 유일한 방법은 상태 설정 함수를 사용해 상태를 업데이트하는 것입니다.
- 컴포넌트가 재렌더링될 때, props가 변경되지 않아도 모든 자식 컴포넌트를 다시 렌더링합니다.
- `React.memo`를 컴포넌트에 적용하여 최적화할 수 있습니다. 이렇게 하면 props 중에 하나라도 변경될 때 다시 렌더링합니다.

:::info 왜 memoization이 기본 동작이 아닌가요?
`React.memo`의 동작이 기본 동작이 아닌지 궁금할 것입니다.
다시 렌더링을 할 필요가 없는 컴포넌트의 렌더링을 건너뛰면 성능을 확실히 향상시킬 수 있습니다.

개발자로서 재렌더링에 드는 비용을 과대평가하는 경향이 있습니다. `Decoration` 컴포넌트의 재렌더링시에 드는 시간은 정말 빠릅니다.
컴포넌트에 props는 많지만 자식 컴포넌트가 많지 않은 경우에는, props 변경 여부를 확인하는 비용이 재렌더링보다 더 들 수도 있습니다.

따라서 공식적인 추천은 성능적인 이슈를 해결하기 위해 `React.memo`를 필요할 때 사용하는 것입니다. 섣부른 최적화를 하지 않도록 주의해야 합니다.
:::

<details>
<summary>섹션 요약</summary>

- `React.memo`는 컴포넌트를 감싸서 props가 변경되지 않으면 재렌더링을 건너뛰는 memoization 기법이다.
- props가 변경된 컴포넌트만 다시 렌더링되고, 변경되지 않은 컴포넌트는 이전 스냅샷을 재사용한다.
- memoization이 기본 동작이 아닌 이유는 props 비교 비용이 재렌더링보다 클 수 있기 때문이며, 성능 이슈가 있을 때만 사용하는 것이 권장된다.

</details>

## 🌃 useMemo Hook

앞에서 `React.memo`를 활용하여 컴포넌트를 memoization 하는 방법을 살펴봤습니다. `React.memo`는 props와 상태가 변경되었을 때만 다시 렌더링됩니다.

이 섹션에서는 `useMemo`를 활용한 다른 방식의 memoization 방법을 살펴보겠습니다.

`useMemo`의 핵심 아이디어는 렌더링 사이에 계산된 값들을 "기억"하는 것입니다.

일반적으로 이 hook을 성능 최적화를 위해 사용합니다. 이 hook은 다르지만 연관되어 있는 두 가지 방식으로 사용됩니다.

1. 렌더링 시에 끝내야 하는 작업의 양을 줄일 수 있습니다.
2. 컴포넌트가 다시 렌더링되는 횟수를 줄일 수 있습니다.

이 두 가지 방식을 살펴보겠습니다.

### 🎆 사용 예시1: 무거운 계산

0에서 사용자가 입력한 `selectedNum` 사이에 소수를 찾는 도구를 만든다고 가정해보겠습니다. 소수는 1과 자기 자신으로만 나누어지는 숫자입니다.

아래는 이 도구를 구현한 코드입니다.

```jsx
function isPrime(n) {
  const max = Math.ceil(Math.sqrt(n));

  if (n === 2) {
    return true;
  }

  for (let counter = 2; counter <= max; counter++) {
    if (n % counter === 0) {
      return false;
    }
  }

  return true;
}

function App() {
  const [selectedNum, setSelectedNum] = useState(100);

  const allPrimes = [];
  for (let counter = 2; counter < selectedNum; counter++) {
    if (isPrime(counter)) {
      allPrimes.push(counter);
    }
  }

  return (
    <>
      <form>
        <label htmlFor="num">Your number:</label>
        <input
          id="num"
          type="number"
          value={selectedNum}
          onChange={(event) => {
            const num = Math.min(100_000, Number(event.target.value));
            setSelectedNum(num);
          }}
        />
      </form>
      <p>
        There are {allPrimes.length} prime(s) between 1 and {selectedNum}:{' '}
        <span className="prime-list">{allPrimes.join(', ')}</span>
      </p>
    </>
  );
}
```

위 코드에서 `selectedNum` 하나의 상태만 있습니다. _for_ 루프를 사용하면 0부터 `selectedNum` 사이의 모든 소수들을 직접 계산합니다.
사용자는 `selectedNum`을 변경할 수 있습니다.

예시 코드는 많은 수의 계산을 합니다. 만약 사용자가 큰 `selectedNum`을 입력하면, 수만 개의 숫자를 하나씩 살펴보면서 각각이 소수인지 확인해야 합니다.
사실 더 효율적인 소수 계산 방법이 있지만, 계산량이 많은 것은 사실입니다.

사실 계산을 완전히 피할 수는 없습니다. 최소 한 번은 해야 하고, 사용자가 새로운 숫자를 입력했을 때마다 새로 계산해야 합니다. 하지만 이 작업을 불필요하게 반복하게 되면 성능 문제가 발생할 수 있습니다.

예를 들어, 위 코드에서 시간까지 보여줘야 한다고 가정해보겠습니다.

그렇게 되면 `selectedNum`과 `time` 두 개의 상태를 가지게 됩니다. 매초마다 `time` 변수가 현재 시간으로 업데이트되며 이 값은 화면에 시간을 나타내기 위해 사용됩니다.

문제는 여기서 발생합니다. 이 상태 중 하나라도 변경된다면, 컴포넌트는 다시 렌더링되고 비용이 많이 드는 계산도 다시 실행됩니다.
`time`이 매초마다 변경되기 때문에 사용자가 입력한 숫자를 바꾸지 않아도 소수의 배열이 다시 생성됩니다.

자바스크립트는 싱글 스레드입니다. 따라서 소수를 계산하는 코드가 매초마다 실행되면 단일 스레드는 매우 바쁘게 돌아갑니다. 이는 사용자가 다른 작업을 동시에 수행하려고 할 때, 특히 저사양 기기에서 애플리케이션이 느려질 수 있음을 의미합니다.

하지만 만약 이 계산을 "건너뛸 수" 있다면 어떻게 될까요? 이미 계산된 소수의 목록이 있다면, 매번 계산하는 것보다는 재사용하는 게 어떨까요?

`useMemo`를 사용하면 가능합니다.

```jsx
const allPrimes = useMemo(() => {
  const result = [];

  for (let counter = 2; counter < selectedNum; counter++) {
    if (isPrime(counter)) {
      result.push(counter);
    }
  }

  return result;
}, [selectedNum]);
```

`useMemo`는 두 개의 인자를 받습니다.

1. 콜백 함수로 감싸진 계산 로직
2. 의존성 배열

컴포넌트가 최초로 렌더링될 때 즉 마운트되는 중에, 리액트는 이 함수를 호출하여 모든 소수들을 계산합니다.
이 함수가 반환하는 것이 무엇이든 `allPrimes`에 할당됩니다.

하지만 그 이후의 렌더링에서는 리액트는 선택을 합니다.

1. 함수를 다시 실행하여 값을 재계산 함
2. 마지막 계산으로부터 나온 값을 재사용 함

위 두 선택지를 결정하기 위해 리액트는 의존성 배열을 봅니다. 이전 렌더링과 비교했을 때 의존성 배열에 있는 값들이 변경되었다면 함수를 재실행하여 값을 계산합니다.
그렇지 않으면 해당 작업을 모두 건너뛰고 이전에 계산한 값을 재사용합니다.

`useMemo`는 작은 캐시와 같고, 의존성 배열은 캐시 무효화 전략과 같습니다.

위 코드에서는 `selectedNum`이 변경될 경우에만 소수의 목록을 재계산합니다.
컴포넌트가 다른 이유(`time` 상태 변수가 변경될 때)로 다시 렌더링될 때, `useMemo`는 함수를 무시하고 캐시된 값을 전달합니다.

:::info 다른 hook과의 유사한 점
`useMemo`의 구조가 `useEffect`와 유사함을 알 수 있습니다! 두 개의 hook 모두 callback 함수와 의존성 배열을 인자로 받습니다.

주요한 차이점은 `useMemo`의 경우 렌더링 도중에 값을 계산하기 위해 사용된다는 것입니다. 하지만 `useEffect`의 경우 리액트의 상태를 외부 시스템과 동기화하기 위해 렌더링 후에 실행합니다.

`useMemo`가 `React.memo` 함수와 비슷한 이름을 가졌다는 것을 눈치채셨을 겁니다.

이는 우연이 아닙니다. 사실 두 개 모두 비슷한 동작을 합니다.

- `React.memo`는 컴포넌트 렌더링의 결과를 기억해서 props나 상태가 변경될 때만 다시 실행합니다.
- `React.useMemo`는 계산의 결과를 기억해서 의존성 배열에 있는 값들이 변경될 때 다시 실행합니다.
  :::

아래는 `useMemo`를 사용한 코드 전체입니다.

```jsx
function App() {
  const [selectedNum, setSelectedNum] = React.useState(100);
  const time = useTime();

  const allPrimes = React.useMemo(() => {
    const result = [];

    for (let counter = 2; counter < selectedNum; counter++) {
      if (isPrime(counter)) {
        result.push(counter);
      }
    }

    return result;
  }, [selectedNum]);

  return (
    <>
      <p className="clock">{format(time, 'hh:mm:ss a')}</p>
      <form>
        <label htmlFor="num">Your number:</label>
        <input
          id="num"
          type="number"
          value={selectedNum}
          onChange={(event) => {
            // To prevent computers from exploding,
            // we'll max out at 100k
            const num = Math.min(100_000, Number(event.target.value));

            setSelectedNum(num);
          }}
        />
      </form>
      <p>
        There are {allPrimes.length} prime(s) between 1 and {selectedNum}:{' '}
        <span className="prime-list">{allPrimes.join(', ')}</span>
      </p>
    </>
  );
}

function isPrime(n) {
  const max = Math.ceil(Math.sqrt(n));

  if (n === 2) {
    return true;
  }

  for (let counter = 2; counter <= max; counter++) {
    if (n % counter === 0) {
      return false;
    }
  }

  return true;
}
```

### 🎆 사용 예시2: 참조 유지하기

앞에서 `useMemo`가 비용이 비싼 계산을 캐싱함으로써 성능을 향상시키는 것을 봤습니다. 하지만 `useMemo`는 또 다른 방식으로 사용할 수도 있습니다.

아래의 코드에서 `Boxes` 컴포넌트를 만들었습니다. 이 컴포넌트는 장식용으로 사용할 수 있는 다채로운 색상의 박스들을 표시합니다.

또한 box와 관련 없는 `name`이라는 상태를 추가했습니다.

```jsx
function Boxes({ boxes }) {
  return (
    <div className="boxes-wrapper">
      {boxes.map((boxStyles, index) => (
        <div key={index} className="box" style={boxStyles} />
      ))}
    </div>
  );
}

const PureBoxes = React.memo(Boxes);

function App() {
  const [name, setName] = React.useState('');
  const [boxWidth, setBoxWidth] = React.useState(1);

  const id = React.useId();

  // Try changing some of these values!
  const boxes = [
    { flex: boxWidth, background: 'hsl(345deg 100% 50%)' },
    { flex: 3, background: 'hsl(260deg 100% 40%)' },
    { flex: 1, background: 'hsl(50deg 100% 60%)' },
  ];
  return (
    <>
      <PureBoxes boxes={boxes} />

      <section>
        <label htmlFor={`${id}-name`}>Name:</label>
        <input
          id={`${id}-name`}
          type="text"
          value={name}
          onChange={(event) => {
            setName(event.target.value);
          }}
        />
        <label htmlFor={`${id}-box-width`}>First box width:</label>
        <input
          id={`${id}-box-width`}
          type="range"
          min={1}
          max={5}
          step={0.01}
          value={boxWidth}
          onChange={(event) => {
            setBoxWidth(Number(event.target.value));
          }}
        />
      </section>
    </>
  );
}
```

`Boxes` 컴포넌트는 `React.memo`로 감싸졌기 때문에 순수합니다. 다시 말해, 이 컴포넌트는 props나 상태가 변경될 때만 다시 렌더링됩니다.

하지만 사용자가 이름을 바꾸면 `PureBoxes` 컴포넌트 또한 다시 렌더링됩니다.

왜 이럴까요? `React.memo`가 재렌더링을 막아주지 않는걸까요?

`PureBoxes` 컴포넌트는 `boxes`라는 하나의 props만 받고 있으며 매 렌더링마다 항상 같은 값을 받습니다.
항상 red, purple, yellow를 가집니다. `boxWidth` 상태 변수가 `boxes` 배열에 영향을 미치지만 변경하지 않았습니다!

문제는 여기서 발생합니다. 리액트가 매번 재렌더링을 할 때마다, 새로운 배열을 생성합니다. 값의 관점에서는 같지만 참조값의 관점에서는 다릅니다.

리액트는 잠시 잊고 순수한 자바스크립트 객체를 떠올려보겠습니다.

```js
function getNumbers() {
  return [1, 2, 3];
}

const firstResult = getNumbers();
const secondResult = getNumbers();

console.log(firstResult === secondResult);
```

어떻게 생각하시나요? `firstResult`와 `secondResult`가 같을까요?

어떤 의미에서는 그렇습니다. 두 개의 값은 모두 `[1, 2, 3]`으로 동일한 구조를 갖습니다. 하지만 `===` 연산자가 확인하는 것은 그것이 아닙니다.

대신에 `===` 연산자는 두 개의 표현식이 같은지 여부를 확인합니다.

배열이나 객체의 경우 겉모습이 같다고 해서 충분한 것은 아닙니다. 두 값이 같으려면 같은 메모리 주소를 가리켜야 합니다.

`getNumbers` 함수를 호출할 때마다, 새로운 배열을 생성하게 되고 이 배열은 메모리의 다른 위치에 저장됩니다.
만약 여러 번 호출한다면 여러 개의 복사본을 메모리에 저장하는 것입니다.

문자열이나, 숫자, 불리언 타입과 같은 간단한 데이터 타입들은 값으로 비교될 수 있습니다. 하지만 배열이나 객체의 경우 참조값으로 비교해야 합니다.

이 내용을 염두에 두고 리액트로 다시 돌아가 보겠습니다.
`PureBoxes` 컴포넌트는 자바스크립트 함수입니다. 렌더링을 할 때 함수를 실행합니다.

```jsx
// Every time we render this component, we call this function...
function App() {
  // ...and wind up creating a brand new array...
  const boxes = [
    { flex: boxWidth, background: 'hsl(345deg 100% 50%)' },
    { flex: 3, background: 'hsl(260deg 100% 40%)' },
    { flex: 1, background: 'hsl(50deg 100% 60%)' },
  ];

  // ...which is then passed as a prop to this component!
  return <PureBoxes boxes={boxes} />;
}
```

`name` 상태가 변경될 때, `App` 컴포넌트는 다시 렌더링되면서 모든 코드가 다시 실행됩니다. 이 때 `boxes` 배열도 다시 생성되며 `PureBoxes` 컴포넌트로 전달됩니다.

그리고 `PureBoxes` 컴포넌트는 새로운 배열을 props로 받았기 때문에 다시 렌더링됩니다.

`boxes`의 배열 구조는 변경되지 않았지만 동일한 값은 아닙니다. 리액트가 아는 것은 `boxes` props에 이전과는 다른 배열이 전달되었다는 것입니다.

이 문제를 해결하기 위해서 `useMemo`를 사용할 수 있습니다.

```js
const boxes = useMemo(() => {
  return [
    { flex: boxWidth, background: 'hsl(345deg 100% 50%)' },
    { flex: 3, background: 'hsl(260deg 100% 40%)' },
    { flex: 1, background: 'hsl(50deg 100% 60%)' },
  ];
}, [boxWidth]);
```

앞서 살펴본 소수 계산과는 달리 비용이 비싼 계산을 걱정할 필요는 없습니다. `useMemo`를 사용한 목적은 배열의 참조값을 저장하기 위함입니다.

`boxWidth`를 의존성 배열로 추가했는데 이 값이 바뀌면 `PureBoxes` 컴포넌트가 다시 렌더링되어야 하기 때문입니다.

지금까지 살펴본 내용을 그림으로 표현해보겠습니다.

## 🌃 useCallback Hook

`useCallback`은 `useMemo`와 동일하게 "참조값을 유지"시켜 줍니다. 하지만 배열이나 객체가 아니라 함수에 대한 참조값을 유지시켜 줍니다.

배열, 객체와 비슷하게 함수도 값이 아닌 참조값으로 비교합니다.

```js
const functionOne = function () {
  return 5;
};

const functionTwo = function () {
  return 5;
};

console.log(functionOne === functionTwo); // false
```

이 말은 컴포넌트 안에 정의한 함수들이 매 렌더링마다 새로 생성된다는 뜻입니다.

아래의 코드를 살펴보겠습니다.

```jsx
function MegaBoost({ handleClick }) {
  console.log('Render MegaBoost');

  return (
    <button className="mega-boost-button" onClick={handleClick}>
      MEGA BOOST!
    </button>
  );
}

export default React.memo(MegaBoost);

function App() {
  const [count, setCount] = React.useState(0);

  function handleMegaBoost() {
    setCount((currentValue) => currentValue + 1234);
  }

  return (
    <>
      Count: {count}
      <button
        onClick={() => {
          setCount(count + 1);
        }}
      >
        Click me!
      </button>
      <MegaBoost handleClick={handleMegaBoost} />
    </>
  );
}
```

위 예시는 흔한 counter 애플리케이션이지만, "Mega Boost" 버튼이 추가되었습니다. 이 버튼은 급할 때 일반 버튼을 여러번 클릭하지 않고도 카운터를 큰 수로 증가시킵니다.

`MegaBoost` 컴포넌트는 `React.memo`로 감싸져 있기 때문에 순수한 컴포넌트입니다. 이 컴포넌트는 `count`를 props로 받지 않지만, `count`가 변경될 때마다 다시 렌더링됩니다!

문제는 매 렌더링마다 새로운 함수를 생성하기 때문에 발생합니다. 만약 3번 렌더링한다면, 3개의 개별 `handleMegaBoost` 함수를 생성하게 되고 이는 `React.memo`를 무력화 시킵니다.

앞 섹션에서 살펴본 `boxes` 배열과 동일한 문제이지만 함수를 props로 전달했다는 점이 다릅니다.

`useMemo`에서 살펴봤던것을 사용하면 다음과 같이 문제를 해결할 수 있습니다.

```jsx
const handleMegaBoost = useMemo(() => {
  return function () {
    setCount((currentValue) => currentValue + 1234);
  };
}, []);
```

배열을 반환하는 대신에, 함수를 반환했습니다. 이제 함수는 `handleMegaBoost` 변수에 저장되었습니다.

이렇게 해도 동작하지만.... 다음과 같이 하는게 더욱 편합니다.

```jsx
const handleMegaBoost = useCallback(() => {
  setCount((currentValue) => currentValue + 1234);
}, []);
```

`useCallback`은 `useMemo`와 동일한 목적을 수행하지만 함수를 위해 설계되었습니다. 함수를 전달하면 렌더링간에 함수를 기억합니다.

달리 말하면, 아래의 두 표현식은 정확히 동일한 결과를 반환합니다.

```jsx
React.useCallback(function helloWorld() {}, []);

React.useMemo(() => function helloWorld() {}, []);
```

사실, `useCallback`은 문법적 설탕입니다.

:::info 상태 설정함수 콜백
앞서 살펴본 예시에서 상태를 업데이트하기 위해 setter에 콜백함수를 전달했습니다.

왜 이 방법을 사용해야 할까요? 일반적인 방법을 어떻게 되는지 알아보겠습니다.

```jsx
const handleMegaBoost = React.useCallback(() => {
  setCount(count + 1234);

  // ⚠️ ESLint warning ⚠️
  // `count` is missing from the dependency array
}, []);
```

`count`에 접근하고 있지만 의존성 배열에 이 값을 추가하지 않았습니다. Effect에서 보았듯이 `count`는 오래된 상태(stale)가 될 것 입니다.
실제 `count` 변수가 변해도 함수 내부에 있는 `count`는 항상 `0`입니다.

`count`를 의존성 배열에 추가하면 이 문제를 해결할 수 있습니다.

```jsx
const handleMegaBoost = React.useCallback(() => {
  setCount(count + 1234);
}, [count]);
```

이제 동작합니다만, `count`가 변경될 때마다 `handleMegaBoost` 함수도 다시 생성됩니다. 즉 `count`가 변경될 때 `MegaBoost` 컴포넌트는 다시 렌더링되며 이는 우리가 피하고자 했던 상황입니다.

Setter 함수에 콜백함수를 전달하면 `count`가 변경되어도 새로운 함수를 생성하지 않고 항상 최신 값에 접근할 수 있게 해줍니다.

:::
