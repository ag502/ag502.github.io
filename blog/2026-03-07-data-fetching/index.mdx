데이터를 불러오는 것은 대부분의 웹 애플리케이션에서 매우 중요한 요소입니다. 대부분의 웹 애플리케이션은 데이터베이스에서 불러온 데이터를 바탕으로 사용자 인터랙션을 처리하기 위해 존재합니다.

- 검색 엔진은 수억 개의 데이터 중에서 가장 적합한 결과를 필터링 해주는 거대한 데이터베이스입니다.
- 슬랙 같은 채팅 애플리케이션은 데이터베이스에 저장된 메시지를 주고 받을 수 있습니다.
- 소셜 네트워크는 데이터베이스로부터 피드를 제공하며 데이터베이스에 있는 피드들을 업데이트함으로써 상호 작용을 할 수 있게 합니다.

이 글에서는 리액트에서 네트워크 요청을 어떻게 보내는지 알아보겠습니다.

## 🌃 이벤트를 통해서 불러오기

개발자 포트폴리오 사이트를 만들어본 경험이 있다면 *연락처 폼*을 구현해 본 적이 있을 겁니다. 이 폼들은 웹 애플리케이션에서 흔하게 볼 수 있습니다.

어떻게 구현할 수 있는지 알아보겠습니다.

```jsx live
function ContactForm() {
  const ENDPOINT = '';
  const [email, setEmail] = React.useState('');
  const [message, setMessage] = React.useState('');

  const id = React.useId();
  const emailId = `${id}-email`;
  const messageId = `${id}-message`;

  async function handleSubmit(event) {
    event.preventDefault();

    const response = await fetch(ENDPOINT, {
      method: 'POST',
      body: JSON.stringify({ email, message }),
    });

    const json = await response.json();
    console.log(json);
  }

  return (
    <form onSubmit={handleSubmit}>
      <div className="row">
        <label htmlFor={emailId}>Email</label>
        <input
          required={true}
          id={emailId}
          type="email"
          value={email}
          onChange={(event) => {
            setEmail(event.target.value);
          }}
        />
      </div>
      <div className="row">
        <label htmlFor={messageId}>Message</label>
        <textarea
          required={true}
          id={messageId}
          value={message}
          onChange={(event) => {
            setMessage(event.target.value);
          }}
        />
      </div>
      <div className="button-row">
        <span className="button-spacer" />
        <button>Submit</button>
      </div>
    </form>
  );
}
```

- 위 코드에서 이메일과 메시지에 대하여 제어 인풋을 추가했습니다.
- form의 `onSubmit` 속성에 이벤트 함수를 전달하였고, 해당 함수에 `event.preventDefault()`를 호출하여 기본 동작을 막았습니다.
  해당 함수에서 데이터 제출을 처리하기 위해 `fetch` 함수를 사용했습니다.
- `fetch` 함수는 promise 기반의 함수이기 때문에 이벤트 함수를 async 함수로 변경했습니다. 따라서 응답을 기다릴 수 있습니다.

위 코드는 이벤트 기반의 네트워크 요청을 할 때 기본적인 방법이지만 UX적인 관점에서 추가해야 할 코드가 많습니다.

### 🎆 Loading, success, error 상태 처리

네트워크 요청을 보낼 때, UI를 아래의 3가지 상태에 따라 업데이트 해야하는 경우가 있습니다.

- Loading
- Success
- Error

우선 `status` 상태는 4가지의 값을 가질 수 있습니다.

- `idle`
- `loading`
- `success`
- `error`

Form이 제출되면 상태는 `loading`으로 변경됩니다.

요청이 성공했다는 건 어떻게 알까요? 상태 코드를 사용할 수도 있지만 신뢰성이 떨어질 수 있기 때문에 JSON의 결과를 보겠습니다.
만약 `json.ok`의 값이 truthy한 값이면 상태를 `success`로 변경하고 그렇지 않으면 `error`로 변경합니다.

상태가 `loading`이면 UI를 두 가지 방향으로 변경할 것입니다.

- 모든 인풋 필드와 버튼이 비활성화 됩니다. 이는 무언가 발생한다는 힌트인 동시에 중복 요청을 방지하는 역할을 합니다.
- 버튼의 텍스트가 "Submit"이나 "Submitting..."으로 변경됩니다.

상태가 `success`일 때, 몇 가지 선택지가 있습니다. Form을 성공 메시지로 교체하거나, form 아래에 작은 메시지를 보여줄 수 있습니다. 만약 form 아래에 메시지를 보여주는 방법을 선택했다면, 메시지를 리셋하는 기능이 필요합니다.

`error` 상태에서는 form 아래에 오류 메시지를 보여줘야 합니다.

위 계획을 구현한 코드는 아래와 같습니다.

```jsx live
function ContactForm() {
  const ENDPOINT = '';

  const [email, setEmail] = useState('');
  const [message, setMessage] = useState('');

  // idle | loading | success | error
  const [status, setStatus] = useState('idle');

  const id = React.useId();
  const emailId = `${id}-email`;
  const messageId = `${id}-message`;

  async function handleSubmit(event) {
    event.preventDefault();

    setStatus('loading');

    const response = await fetch(ENDPOINT, {
      method: 'POST',
      body: JSON.stringify({
        email,
        message,
      }),
    });
    const json = await response.json();

    if (json.ok) {
      setStatus('success');
      setMessage('');
    } else {
      setStatus('error');
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <div className="row">
        <label htmlFor={emailId}>Email</label>
        <input
          required={true}
          disabled={status === 'loading'}
          id={emailId}
          type="email"
          value={email}
          onChange={(event) => {
            setEmail(event.target.value);
          }}
        />
      </div>
      <div className="row">
        <label htmlFor={messageId}>Message</label>
        <textarea
          required={true}
          disabled={status === 'loading'}
          id={messageId}
          value={message}
          onChange={(event) => {
            setMessage(event.target.value);
          }}
        />
      </div>
      <div className="button-row">
        <span className="button-spacer" />
        <button disabled={status === 'loading'}>
          {status === 'loading' ? 'Submitting…' : 'Submit'}
        </button>
      </div>
      {status === 'success' && <p>Message sent!</p>}
      {status === 'error' && <p>Something went wrong!</p>}
    </form>
  );
}
```

:::info 예측하지 못한 오류 잡기
위 코드에서 JSON 응답을 확인함으로써 예측할 수 있는 오류를 처리했습니다. 하지만 예측하지 못한 오류가 발생할 수 있습니다.
서버가 죽는 상황이 그 예시입니다.

이 상황을 처리하기 위해 fetch 요청을 `try`/`catch` 문으로 감싸야 합니다.

```js
async function handleSubmit(event) {
  event.preventDefault();

  setStatus('loading');

  try {
    const response = await fetch(ENDPOINT, {
      method: 'POST',
      body: JSON.stringify({
        email,
        message,
      }),
    });
    const json = await response.json();

    if (json.ok) {
      setStatus('success');
      setMessage('');
    } else {
      setStatus('error');
    }
  } catch (err) {
    setStatus('error');
  }
}
```

:::

<details>
<summary>섹션 요약</summary>

- 이벤트 기반 데이터 페칭은 form의 `onSubmit` 핸들러에서 `fetch`를 호출하고, `event.preventDefault()`로 기본 동작을 막는 방식으로 구현한다
- `status` 상태(`idle`, `loading`, `success`, `error`)를 관리하여 로딩 중 인풋 비활성화, 성공/오류 메시지 표시 등 UX를 개선할 수 있다
- 예측 가능한 오류는 JSON 응답으로, 예측하지 못한 오류는 `try`/`catch`로 처리한다

</details>

## 🌃 마운트시에 데이터 불러오기

이전 섹션에서 form 제출과 같은 이벤트가 발생할 때 네트워크 요청을 보내는 방법을 살펴봤습니다. 하지만 화면에 데이터를 표시하기 위해 렌더링 시에 불러와야 하면 어떻게 할까요?

날씨 앱을 예시로 들어보겠습니다. 현재 온도가 몇도인지를 사용자에게 표시해주고 싶습니다.

이 문제는 생각보다 까다롭습니다. 마운트시에 데이터를 불러오는 것과 이벤트 발생 시에 불러오는 것의 차이는 사소해 보일 수 있지만 실제로는 훨씬 더 복잡한 문제를 야기할 수 있습니다.

첫 번째로, 사용하기가 까다롭습니다. 마운트 시에 데이터를 불러오려면 일반적으로 `useEffect` hook을 사용하지만 effect 내부에서 async/await를 사용하기 위해서는 몇 가지 지켜야 할 점이 있습니다(아래에서 살펴보겠습니다).

또한 이 문제를 견고하고 프로덕션에 적용하기 위해서 몇 가지 고려해야 할 사항도 있습니다.

- 응답을 캐싱처리하여 여러 컴포넌트에서 재사용 가능하게 해야 합니다.
- 데이터가 오래된 상태(stale)가 되지 않도록 재검증해야 합니다.

이런 문제들 때문에, 이를 도와주는 도구를 사용하는 것이 커뮤니티에서는 표준이 되었습니다. 예를 들어 리액트 문서에서는 React Query나 SWR과 같은 라이브러리를 제안하고 있습니다.
이 글에서는 SWR에 대해 살펴보겠습니다.

### 🎆 SWR 살펴보기

아래는 간단한 애플리케이션입니다.

```jsx
const ENDPOINT = '';

async function fetcher(endpoint) {
  const response = await fetch(endpoint);
  const json = await response.json();

  return json;
}

function App() {
  const { data, error } = useSWR(ENDPOINT, fetcher);

  console.log(data, error);

  return (
    <p>
      Current temperature:
      {typeof data?.temperature === 'number' && (
        <span className="temperature">{data.temperature}°C</span>
      )}
    </p>
  );
}
```

위 코드에서 `useSWR` 함수를 사용하고 있습니다. 이 함수는 커스텀 hook입니다.

```js
const { data, error } = useSWR(ENDPOINT, fetcher);
```

`useSWR`은 두 개의 인자를 받습니다.

- 고유한 키
  보통 요청을 보낼 endpoint 주소입니다.
- fetcher 함수

보통 사람들은 SWR과 같은 라이브러리들이 네트워크 요청을 보낸다고 생각하지만 이는 잘못된 개념입니다. 이런 종류의 라이브러리를 사용하든 하지 않든, fetch 함수를 작성해야 합니다.
다시 말해 데이터를 불러오는 것에 관해서는 프레임워크에 비의존적입니다. SWR은 요청을 언제 보내고 응답값을 어떻게 처리할지를 결정합니다.

아래는 fetcher 함수입니다.

```js
async function fetcher(endpoint) {
  const response = await fetch(endpoint);
  const json = await response.json();

  return json;
}
```

`fetcher` 함수가 무엇을 반환하든 useSWR의 `data` 변수에 저장됩니다. 이 경우 `json`의 형태는 아래와 같습니다.

```json
{
  "ok": true,
  "temperature": 20
}
```

`App` 컴포넌트에서 `useSWR`로부터 `data`와 `error`를 추출해 렌더링에 이 변수들을 사용합니다. 만약 `data`가 존재하면 요청이 성공했다는 의미이고, UI에서 이 값을 쓸 수 있습니다.

`useSWR`이 fetcher 함수를 호출할 때 첫 번째 파라미터로 키값을 전달합니다. 위 예시에서 `ENDPOINT` 변수를 직접 사용할 수 있기 때문에 불필요해 보이지만, 핵심은 fetcher 함수는 범용적으로 설계되었다는 점입니다.
"helper.js" 파일에 `fetcher` 함수를 정의해서 마운트 시의 네트워크 요청에 재사용할 수 있습니다.

Fetcher 함수를 우리가 직접 작성하는 것이 꽤 번거로워 보입니다. 그렇다면 이 라이브러리들이 제공하는 장점은 뭘까요?

첫 번째 장점은 "재검증(revalidation)"입니다. 이름에서 알 수 있듯이 SWR은 Stale While Revalidate의 약자입니다.

네트워크 요청은 마운트됨과 동시에 발생하며 값을 가져옵니다. 또한 전략적인 시점에만 요청이 발생하여 데이터가 오래된 상태(stale)가 되지 않도록 합니다.

예를 들어, 탭을 전환하는 상황을 생각해봅시다. 한 탭에서 일정 시간 머무르다 다른 탭으로 전환할 때 `useSWR`은 자동으로 네트워크 요청을 보내어 새로운 값을 가져옵니다. 이를 통해 사용자는 최신 온도 데이터를 볼 수 있습니다.

이와 같은 동작은 재검증(revalidating)이라고 합니다. 값이 변경되면 즉시 오래된 값을 새로운 값으로 변경합니다.

하지만 요청이 진행 중이라면, 오래된 값이 보이게 됩니다. 오래된 값(stale)은 재검증(revalidating)이 일어나는 동안 보입니다.

이 동작은 커스터마이즈 할 수 있습니다.

### 🎆 Loading, error 상태

SWR을 이용해 loading, error 상태를 처리하는 방법을 알아보겠습니다.

#### Loading

`useSWR`은 `data` 외에도 현재 요청이 진행 중인지 여부를 파악할 수 있는 값을 제공합니다. `isLoading` 키로 확인할 수 있습니다.

```js
const { data, isLoading } = useSWR(ENDPOINT, fetcher);
```

`isLoading`은 불리언 타입입니다. 초기값은 `true`이고, 요청이 완료되면 `false`로 변경됩니다.

이 값을 loading UI를 조건부로 렌더링하기 위해 사용할 수 있습니다.

```jsx
function App() {
  const { data, isLoading } = useSWR(ENDPOINT, fetcher);

  if (isLoading) {
    return <p>Loading…</p>;
  }

  return (
    <p>
      Current temperature:
      {typeof data.temperature === 'number' && (
        <span className="temperature">{data.temperature}°C</span>
      )}
    </p>
  );
}
```

#### Error

SWR에는 에러를 처리할 수단이 없습니다.

네트워크 요청을 다루는 주체는 우리임을 명심하세요. `fetcher` 함수가 데이터를 가져오는 동시에 SWR에 전달하는 역할을 담당합니다. 만약 에러를 처리하고 싶다면 에러를 throw 해야 합니다.

```js
async function fetcher(endpoint) {
  const response = await fetch(endpoint);
  const json = await response.json();

  if (!json.ok) {
    throw json;
  }

  return json;
}
```

위 `fetcher` 함수를 사용하면 오류가 발생했을 때 `data`는 undefined가 되고, `error`는 `fetcher` 함수에서 throw한 값이 됩니다.

상태값을 반영한 최종 결과물은 아래와 같습니다.

```jsx
const ENDPOINT = '';

async function fetcher(endpoint) {
  const response = await fetch(endpoint);
  const json = await response.json();

  if (!json.ok) {
    throw json;
  }

  return json;
}

function App() {
  const { data, isLoading, error } = useSWR(ENDPOINT, fetcher);

  if (isLoading) {
    return <p>Loading…</p>;
  }

  if (error) {
    return <p>Something has gone wrong</p>;
  }

  return (
    <p>
      Current temperature:
      {typeof data?.temperature === 'number' && (
        <span className="temperature">{data.temperature}°C</span>
      )}
    </p>
  );
}
```

<summary>섹션 요약</summary>

- 마운트 시 데이터를 불러오는 것은 캐싱, 재검증 등 고려할 사항이 많아 SWR이나 React Query 같은 라이브러리를 사용하는 것이 표준이다
- SWR은 네트워크 요청 자체를 보내지 않으며, 직접 작성한 fetcher 함수를 언제 호출하고 응답을 어떻게 관리할지를 담당한다
- SWR은 탭 전환 등 전략적 시점에 자동으로 재검증하여 데이터가 stale 상태가 되지 않도록 한다
- `isLoading`으로 로딩 상태를 처리하고, fetcher 함수에서 에러를 throw하면 `error` 값으로 에러 상태를 처리할 수 있다

</details>

## 🌃 비동기 effect 살펴보기

SWR을 사용하지 않고 마운트 시에 데이터를 불러와야 한다고 가정해보겠습니다. `fetch` 요청을 `useEffect` 훅 내부에 선언해야 합니다.

대부분의 개발자들은 async/await 구문을 사용하여 아래처럼 작성할 것입니다.

```
useEffect(async () => {
  const response = await fetch(ENDPOINT);
  const json = await response.json();

  setTemperature(json.temperature);
}, []);
```

`await` 키워드를 사용하려면, `async` 함수 내부에서 사용해야 합니다. Effect 함수의 콜백을 async로 변경하면 될 것 같았지만 안타깝게도 그러면 동작하지 않습니다.

Effect의 callback을 비동기로 만드는 것은 안됩니다. 대신에 effect 내부에 새로운 함수를 만들어야 합니다.

```js
React.useEffect(() => {
  // Create an async function within our effect:
  async function runEffect() {
    const response = await fetch(ENDPOINT);
    const json = await response.json();

    setTemperature(json.temperature);
  }

  // Immediately call this function to run the effect:
  runEffect();
}, []);
```

:::info 왜 effect callback은 비동기가 될 수 없을까요?
비동기 함수를 생성하고 바로 실행하는 것이 이상해 보일 수 있습니다. 왜 `useEffect`에 비동기 함수를 전달할 수 없는 걸까요?

비동기 함수에 대해 이해할 사항은 항상 promise를 반환한다는 것입니다.

예를 들어,

```js
async function quickExample() {
  return 5;
}

const result = quickExample();
console.log(result); // Promise
```

`quickExample` 함수는 숫자 `5` 를 반환하지 않습니다. `5`를 resolve하는 promise를 반환합니다. 이는 `await`를 하지 않는 함수를 포함해서 모든 비동기 함수에 해당하는 내용입니다.

리액트는 promise가 아닌 cleanup 함수를 반환해야 합니다.

아래와 같은 effect를 살펴보겠습니다.

```js
React.useEffect(() => {
  // Effect logic: start an interval
  const intervalId = window.setInterval(() => {}, 1000);

  // Cleanup logic: stop the interval:
  return () => {
    window.clearInterval(intervalId);
  };
}, []);
```

리액트가 effect를 실행할 때 cleanup 함수를 전달합니다. 리액트는 의존성이 변경되거나 컴포넌트가 언마운트될 때 cleanup 함수를 실행합니다.

이제 effect 내부에서 비동기 작업을 수행하는 예시를 살펴보겠습니다.

```
React.useEffect(async () => {
  const intervalId = window.setInterval(() => {}, 1000);

  await someLongRunningProcess();

  // Even though we're returning a function, it'll actually
  // return a *promise* that resolves to a function:
  return () => {
    window.clearInterval(intervalId);
  };
});
```

리액트가 이 effect를 실행할 때 cleanup 함수가 아닌 promise를 전달 받습니다.

장기적으로 실행되는 작업이 진행 중일 때 컴포넌트가 언마운트 된다면 어떻게 될까요? cleanup 함수는 실행되지 않습니다. 리액트가 cleanup 함수를 전달받지 못했기 때문입니다.

:::

<details>
<summary>섹션 요약</summary>

- `useEffect`의 콜백을 직접 async로 만들 수 없으며, effect 내부에 별도의 async 함수를 선언한 뒤 즉시 호출해야 한다
- async 함수는 항상 promise를 반환하기 때문에, 콜백을 async로 만들면 리액트가 cleanup 함수 대신 promise를 전달받아 cleanup이 동작하지 않는다

</details>
