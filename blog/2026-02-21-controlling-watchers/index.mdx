## 🌃 Watcher 일시 정지, 재개하기

조건이 충족되지 않을 때 일시적으로 watcher를 비활성화할 필요가 있습니다. 조건이 바뀌면 언제든 다시 활성화할 수 있습니다.

Vue는 watcher를 일시 정지하거나 재개하는 기능을 제공합니다. 이는 오프라인/온라인 상태에서나 로딩 상태일 때 사용하기 적절합니다.

### 🎆 항상 활성화된 watcher의 문제점

```js
import { ref, watch } from '@vue/reactivity';

const doc = ref('');
const isOnline = ref(true);

function saveDoc(text) {
  console.log('Saved:', text);
}

// 이 watcher는 항상 실행됩니다. 심지어 오프라인 환경에서도요.
watch(doc, saveDoc);

// offline 상태로 변경 및 편집
async function demo() {
  doc.value = 'initial content'; // Saved

  isOnline.value = false; // offline 전환
  doc.value = 'offline edit'; // Saved (offline 환경에서는 저장되면 안 됨)

  isOnline.value = true; // online 전환
  doc.value = 'online edit'; // Saved
}
```

문제는 watcher가 오프라인 상태에서도 실행된다는 점입니다. 오프라인 환경에서는 저장이 실패할 것이기 때문에 실행되면 안 됩니다.

### 🎆 일시 정지, 재개 기능 사용하기

Watcher는 동작을 제어하는 함수를 제공합니다.

```js
import { ref, watch } from '@vue/reactivity';

const doc = ref('');
const isOnline = ref(true);

function saveDoc(text) {
  console.log('Saved:', text);
}

// Watcher로부터 반환된 동작 제어 함수
const { pause, resume } = watch(doc, saveDoc);

watch(isOnline, (online) => {
  if (online) {
    resume();
  } else {
    pause();
  }
});
```

이제 자동저장 watcher는 오프라인 상태일 때 자동으로 일시 정지되고 온라인으로 전환될 때 다시 재개됩니다.

위 코드를 아래처럼 간단히 작성할 수도 있습니다.

```js
watch(isOnline, (online) => (online ? resume() : pause()));
```

코드를 실행한 결과는 다음과 같습니다.

```
--- Testing autosave ---
Saved: initial content
Saved: edit 1

--- Going offline ---
Made edit while offline - queued but not saved

--- Coming back online ---
Saved: offline edit
Saved: online edit
```

무슨 일이 일어나는지 살펴보겠습니다.

온라인일 때는 변경된 값이 바로 저장됩니다. 앱이 오프라인 상태에 들어가면 watcher는 일시 정지됩니다.

Watcher가 일시 정지 상태일 때 값을 변경하게 되면 Vue 반응형 시스템이 해당 값들을 큐에 저장합니다. 하지만 저장 함수를 호출하지는 않습니다.

온라인 상태로 전환되면, watcher가 다시 동작하게 됩니다. 그리고 큐에 저장되어있던 값들도 실행됩니다. "offline edit"이 그 결과입니다.

그 후에 온라인 상태에서 발생하는 변경 값은 다시 저장됩니다.

### 🎆 Queue에 저장하는 동작 살펴보기

Vue의 반응형 시스템은 watcher가 일시 정지되었을 때 자동으로 콜백 함수의 호출을 큐에 저장합니다.
변경사항이 유실되는 것이 아니라 잠시 보류되는 것입니다.

큐에 저장하는 방식도 효율적입니다. 일시 정지된 상태에서 동일한 reactive 값이 여러 번 변경되더라도 마지막 변경 사항만 트리거됩니다.

### 🎆 실전에서 활용하기

이 패턴은 아래와 같은 상황에서 매우 유용합니다.

- 네트워크 연결
  오프라인일 때 API 호출을 정지했다가 온라인일 때 다시 호출하게 할 수 있습니다.

- 로딩 상태
  로딩 중일 때 비용이 큰 watcher를 정지할 때 사용할 수 있습니다.

- 사용자 권한 확인
  권한이 없을 때 특정 watcher를 일시 정지시킬 때 사용할 수 있습니다.

- 성능 최적화
  고비용의 작업을 수행할 때에 필수적인 watcher가 아닐 경우 일시 정지시킬 수 있습니다.

<details>
<summary>섹션 요약</summary>

- `watch()`의 반환값에서 `pause()`와 `resume()` 함수를 구조분해 할당으로 가져와 watcher를 일시 정지하거나 재개할 수 있다
- 일시 정지된 동안 발생한 변경사항은 큐에 저장되며, 재개 시 마지막 변경사항만 트리거된다
- 오프라인/온라인 전환, 로딩 상태, 사용자 권한 확인, 성능 최적화 등의 상황에서 활용할 수 있다

</details>

## 🌃 Watcher 정지하기

Watcher는 명시적으로 정지시키거나 effect scope이 정지되지 않으면 계속해서 실행됩니다. 만약 watcher를 동적으로 생성한다면 메모리 누수를 막기 위해 이를 수동으로 제거해줘야 합니다.

모든 watcher는 watcher를 정지시켜 반응형 시스템에서 제거하는 stop 함수를 반환합니다.

### 🎆 메모리 누수 문제

다른 데이터 소스에 대해 watcher를 동적으로 생성하는 모니터링 시스템을 살펴보겠습니다.

```js
import { ref, watch } from '@vue/reactivity';

const serverLoad = ref(0);

function startMonitoring(dataSourceId) {
  console.log(`Starting monitoring for: ${dataSourceId}`);

  // 이 watcher는 계속 실행됩니다.
  watch(serverLoad, (load) => {
    console.log(`Source ${dataSourceId} detected load: ${load}`);
  });
}

function stopMonitoring(dataSourceId) {
  console.log(`Stopping monitoring for: ${dataSourceId}`);
}

// 모니터링 시작
startMonitoring('db-1');
startMonitoring('api-2');
startMonitoring('cache-3');

serverLoad.value = 25; // 3개의 watcher 실행

stopMonitoring('db-1');
stopMonitoring('api-2');

serverLoad.value = 50; // 3개의 watcher 실행
```

위 코드를 실행시키면 아래와 같은 결과가 출력됩니다.

```
--- Adding multiple data sources ---
Starting monitoring for: db-1
Starting monitoring for: api-2
Starting monitoring for: cache-3

--- Server load changes ---
Source db-1 detected load: 25
Source api-2 detected load: 25
Source cache-3 detected load: 25

--- Removing data sources ---
Stopping monitoring for: db-1
Stopping monitoring for: api-2

--- Server load change after "cleanup" ---
Source db-1 detected load: 50    // Still running!
Source api-2 detected load: 50   // Still running!
Source cache-3 detected load: 50
```

위의 예시에서 watcher는 절대 정지되지 않습니다. 이는 메모리 누수를 야기합니다.

### 🎆 해결책: stop 함수 사용하기

Watcher 함수는 stop 함수를 반환합니다.

```js
import { ref, watch } from '@vue/reactivity';

const serverLoad = ref(0);

// stop 함수를 저장하는 자료구조
const watchers = new Map();

function startMonitoring(dataSourceId) {
  console.log(`Starting monitoring for: ${dataSourceId}`);

  // watcher가 반환하는 정지 함수
  const stopWatcher = watch(serverLoad, (load) => {
    console.log(`Source ${dataSourceId} detected load: ${load}`);
  });

  // clean up을 위해 저장
  watchers.set(dataSourceId, stopWatcher);
}

function stopMonitoring(dataSourceId) {
  console.log(`Stopping monitoring for: ${dataSourceId}`);

  const stopWatcher = watchers.get(dataSourceId);

  if (stopWatcher) {
    stopWatcher();
    watchers.delete(dataSourceId);
  }
}
```

이제 `stopMonitoring` 함수를 호출하면 watcher는 정지됩니다.

위 코드를 실행하면 아래와 같이 출력됩니다.

```
--- Adding multiple data sources ---
Starting monitoring for: db-1
Starting monitoring for: api-2
Starting monitoring for: cache-3

--- Server load changes ---
Source db-1 detected load: 25
Source api-2 detected load: 25
Source cache-3 detected load: 25

--- Removing data sources ---
Stopping monitoring for: db-1
Stopping monitoring for: api-2

--- Server load change after "cleanup" ---
Source cache-3 detected load: 50
```

### 🎆 Stop vs Pause 비교하기

Watcher를 정지하는 것과 일시 정지하는 것의 차이를 구별하는 것이 중요합니다.

**Pause/Resume**은 일시적으로 watcher를 비활성화하지만 메모리에는 남겨둡니다. 언제든지 재개할 수 있고, 큐에 저장된 변경사항을 반영할 수도 있습니다.

**Stop**은 watcher를 반응형 시스템에서 완전히 제거합니다. 일단 정지되면, watcher는 다시 시작할 수 없고, 큐에 저장된 것들도 유실됩니다.

### 🎆 Watcher가 자동으로 정지되는 상황

- Component가 언마운트될 때
  Setup 함수 내부에 생성된 watcher는 컴포넌트가 언마운트될 때 자동으로 정지됩니다.

- Effect Scope이 정지될 때
  Effect Scope 내부에 선언된 watcher는 scope이 정지될 때 정지됩니다.

- 수동으로 정지시킬 때
  명시적으로 stop 함수를 호출하면 정지됩니다.

컴포넌트 밖이나 effect scope 밖에서 생성된 watcher는 반드시 명시적으로 정지시켜야 합니다.

<details>
<summary>섹션 요약</summary>

- 동적으로 생성된 watcher를 정지시키지 않으면 메모리 누수가 발생한다
- `watch()`가 반환하는 stop 함수를 호출하여 watcher를 반응형 시스템에서 완전히 제거할 수 있다
- Pause는 일시적 비활성화(재개 가능, 큐 보존)이고, Stop은 완전한 제거(재시작 불가, 큐 유실)이다
- 컴포넌트 언마운트, effect scope 정지 시 자동으로 정지되지만, 그 외에는 반드시 수동으로 정지시켜야 한다

</details>

## 🌃 onCleanup으로 정리하기

사이드 이펙트를 트리거하는 watcher로 작업을 할 때, 메모리 누수를 제거하기 위해 이전의 effect를 정리할 필요가 있습니다. 이 역시 디버깅하기 어려운 버그를 막을 수 있습니다.

적절한 cleanup 없이는 여러 개의 effect들이 중첩됩니다. 이는 애플리케이션의 알 수 없는 버그를 야기합니다.

### 🎆 문제점

아래의 예시는 effect가 적절히 cleanup 되지 않는 케이스입니다.

```js
import { ref, watch } from '@vue/reactivity';

const n = ref(0);

function save() {
  console.log('Saving data...');
}

watch(n, () => {
  // watcher가 실행될 때마다 새로운 timeout 함수가 등록됨
  setTimeout(save, 2000);
});

n.value = 1;
n.value = 2;
n.value = 3;
n.value = 4;
```

위 코드에서 `n`이 변경될 때마다 새로운 timeout이 생성됩니다. 모든 timeout은 2초 후에 실행되며 여러 번 `save` 함수를 호출합니다.

이는 watcher가 실행될 때 이전의 것을 취소하지 않고 새로운 timeout을 생성하기 때문입니다.

### 🎆 해결책

Vue의 watcher 함수는 cleanup을 위해 `onCleanup` 이라는 세 번째 파라미터를 제공합니다.

```js
import { ref, watch } from '@vue/reactivity';

const n = ref(0);

function save() {
  console.log('💾 Saving data...');
}

watch(n, (_new, _old, onCleanup) => {
  const id = setTimeout(save, 2000);

  // 새로운 watcher가 실행되기 전에 timeout clear 하기
  onCleanup(() => clearTimeout(id));
});

n.value = 1;
n.value = 2;
n.value = 3;
n.value = 4;
```

`onCleanup` 함수는 watcher가 다시 실행되기 전에 호출됩니다. 이 함수는 이전에 실행했던 진행 중인 effect를 취소해줍니다.

timeout ID는 클로저로 참조하기 때문에 cleanup 함수는 다음 실행 시에 해당 ID로 접근할 수 있습니다.

### 🎆 동작 원리

Watcher의 콜백 함수는 세 개의 인자를 받습니다.

```js
watch(source, (newValue, oldValue, onCleanup) => {
  //
});
```

`onCleanup` 파라미터는 cleanup을 위한 함수입니다. 이 콜백은 다음 watcher 실행 직전에 호출됩니다. 또한 watcher가 정지될 때도 실행됩니다.

아래는 단계별 실행 순서입니다.

1. Watcher가 실행되고 timeout이 생성됩니다.
2. Timeout ID가 저장됩니다.
3. `onCleanup` 함수가 timeout을 제거하는 함수를 등록합니다.
4. Watcher가 재실행될 때 cleanup 함수가 먼저 실행됩니다.
5. 새로운 timeout이 생성되기 전에 이전 timeout이 제거됩니다.

### 🎆 사용 예시

`onCleanup` 패턴은 취소가 필요한 effect에서 유용합니다.

#### API 요청

```js
watch(searchQuery, (query, _old, onCleanup) => {
  const controller = new AbortController();

  fetchSearchResults(query, { signal: controller.signal }).then((results) =>
    updateUI(results),
  );

  onCleanup(() => controller.abort());
});
```

#### Event Listeners

```js
watch(elementRef, (element, _old, onCleanup) => {
  const handleClick = () => console.log('clicked');
  element.addEventListener('click', handleClick);

  onCleanup(() => element.removeEventListener('click', handleClick));
});
```

#### Subscriptions

```js
watch(userId, (id, _old, onCleanup) => {
  const unsubscribe = subscribeToUser(id, handleUserUpdate);

  onCleanup(unsubscribe);
});
```

핵심은 중첩되는 리소스들을 정리하는 것입니다. 또한 새로운 effect의 실행과 충돌이 되는 리소스도 정리해야 합니다.

<details>
<summary>섹션 요약</summary>

- Watcher 콜백의 세 번째 파라미터 `onCleanup`을 사용하여 다음 실행 전에 이전 effect(timeout, API 요청, 이벤트 리스너 등)를 정리할 수 있다
- cleanup 없이는 effect가 중첩되어 메모리 누수와 예기치 않은 버그가 발생한다
- `onCleanup`에 등록한 함수는 watcher가 재실행되거나 정지될 때 자동으로 호출된다

</details>

## 🌃 onWatcherCleanup으로 decouple된 effect 정리하기

Watcher의 effect를 분리된 헬퍼 함수로 추출하면, `onWatcherCleanup` 함수를 사용하여 cleanup 로직을 분리할 수 있습니다.
이 함수를 사용하면 `onCleanup` 함수를 파라미터로 전달하지 않아도 됩니다.

이렇게 하면 헬퍼 함수의 재사용성이 높아지고 다른 프로젝트나 라이브러리의 이식성도 높아집니다.

### 🎆 문제

복잡한 watcher 로직을 헬퍼 함수로 분리할 때 `onCleanup` 함수도 인자로 넘겨줘야 합니다.

```js
import { ref, watch } from '@vue/reactivity';

function trackBackgroundTask(onCleanup) {
  console.log('🔧 Starting background task helper...');

  const intervalId = setInterval(() => {
    console.log('🔄 Background task is running...');
  }, 1000);

  onCleanup(() => {
    console.log('🧹 Stopping background task helper...');
    clearInterval(intervalId);
  });
}

const n = ref(0);

watch(n, (_new, _old, onCleanup) => {
  trackBackgroundTask(onCleanup);
});
```

이런 결합은 헬퍼 함수가 watcher 내부에서만 호출되어야 하는 제한이 생깁니다. 함수를 다른 곳으로 이동하기도 쉽지 않고 다른 컨텍스트에서 사용하기도 쉽지 않습니다.

### 🎆 해결책

Vue의 `onWatcherCleanup` 함수는 cleanup 파라미터 없이 cleanup을 수행할 수 있게 합니다.

```js
import { ref, watch, onWatcherCleanup } from '@vue/reactivity';

function trackBackgroundTask() {
  console.log('🔧 Starting background task helper...');

  const intervalId = setInterval(() => {
    console.log('📈 Background task running...');
  }, 1000);

  // 헬퍼 함수가 자체 cleanup 로직을 처리합니다.
  onWatcherCleanup(() => {
    console.log('🧹 Cleaning up background task...');
    clearInterval(intervalId);
  });
}

const n = ref(0);

watch(n, () => {
  // 파라미터가 필요 없음
  trackBackgroundTask();
});
```

이제 헬퍼 함수는 watcher의 내부와 완전히 분리되었습니다. Cleanup 로직을 자체적으로 처리하기 때문에 코드를 어디로든 이동시킬 수 있습니다.

### 🎆 동작 원리

`onWatcherCleanup` 함수는 현재 실행 중인 watcher가 effect를 cleanup 할 때 실행되어야 할 함수를 등록합니다.

이 함수는 현재 실행 중인 watcher 컨텍스트를 감지하여 동작합니다. 자동으로 watcher를 감지하여 cleanup 함수를 실행합니다.

```js
watch(someRef, () => {
  // onWatcherCleanup 함수는 현재 실행 중인 watcher가 무엇인지 앎
  onWatcherCleanup(() => {});
});
```

이 방식의 장점은 헬퍼 함수가 watcher의 cleanup 동작 원리를 알 필요가 없다는 것입니다.

### 🎆 주의 사항

`onWatcherCleanup`은 한 가지 주의할 점이 있습니다.

```js
watch(someRef, async () => {
  await someAsyncOperation();

  // 이 코드는 동작하지 않습니다. - 비동기 이후에 context를 잃어버립니다.
  onWatcherCleanup(() => {});
});
```

`await` 이후에 watcher 컨텍스트를 잃어버립니다. 따라서 `onWatcherCleanup`은 어떤 watcher와 연관되어 있는지 알지 못합니다.

비동기 상황에서는 `onCleanup` 함수를 사용해야 합니다.

### 🎆 `onWatcherCleanup`과 `onCleanup`의 사용 예시

#### `onWatcherCleanup`

- 헬퍼 함수 내에서 동기 로직을 추출할 때 사용합니다.
- 재사용 가능한 유틸리티를 만들고 해당 리소스를 직접 제어할 때 사용합니다.
- Watcher 컨텍스트 내에서 동작할 필요가 있는 라이브러리를 만들 때 사용합니다.

#### `onCleanup`

- 비동기 작업 이후에 cleanup 작업을 수행하기 위해 사용합니다.
- Watcher 콜백 내에서 직접 사용하기 위해 씁니다.
- 특정 watcher에 바인딩된 것을 보장하기 위해 사용합니다.

`onWatcherCleanup`을 사용하는 방법은 헬퍼 함수의 모듈화를 도우며, 테스트를 쉽게 할 수 있습니다.

- `onWatcherCleanup`은 현재 실행 중인 watcher 컨텍스트를 자동 감지하여, `onCleanup` 파라미터 없이도 cleanup 로직을 등록할 수 있다
- 헬퍼 함수 내부에서 자체적으로 cleanup을 처리하므로 재사용성과 이식성이 높아진다
- `await` 이후에는 watcher 컨텍스트를 잃어버리므로, 비동기 상황에서는 `onCleanup`을 사용해야 한다
