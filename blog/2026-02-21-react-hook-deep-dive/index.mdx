## 🌃 불변성 다시 살펴보기

이 글에서 리액트의 까다로운 부분중에 하나인 메모이제이션과 `useEffect` hook에 대해 살펴보겠습니다. 이 부분을 제대로 이해하려면 리액트의 핵심 메커니즘인 불변성에 대해 제대로 이해하고 있어야 합니다.

`{name: "Ivy"}`라는 객체를 상태로 들고 있는 `user`라는 변수가 있다고 가정해 보겠습니다.

일반적인 자바스크립트라면 `user.name = 'Ava'`와 같이 객체의 프로퍼티를 변경할 수 있습니다. 하지만 이는 가변입니다! 리액트에서는 이와 같은 방식으로 값을 업데이트 하면 안됩니다.

리액트에서는 `setUser({name: 'Ava'})`를 실행시켜 값을 업데이트 합니다.

해당 함수를 호출할때마다 새로운 객체가 컴퓨터 메모리의 고유한 위치에 저장됩니다.
기존의 `{name: 'Ivy'}` 객체를 변경한것도 아니고, 기존의 객체로 새로운 객체로 변경한것도 아닙니다. 일정기간동안 기존의 객체와 새로운 객체는 모두 존재합니다.

각 스냅샷에는 고유한 `user` 변수가 존재해서 두개의 `user` 변수가 서로 다른 객체를 가리키고 있다고 생각할 수 있습니다.
하지만 실제로는 자바스크립트의 가비지 콜렉터가 오래된 객체를 제거해 메모리가 가득찰 걱정은 하지 않아도 됩니다.

자바스크립트에서 객체는 참조값으로 저장됩니다. 만약 형태가 동일한 두개의 객체 `{name: 'Ava'}`, `{name: 'Ava'}`를 각각 생성한다면 두 객체는 다른 객체입니다.

만약 상태 변수가 여러개라면 어떻게 될까요? 예를 들어, 숫자 배열을 상태로 갖는 `item` 상태가 있다고 가정해보겠습니다.

만약 `setUser()`를 호출해 새로운 스냅샷을 생성해 `user`를 업데이트 한다면, `items` 배열은 재사용됩니다.

각각의 스냅샷은 새로운 `item` 변수를 생성하지만, 변수가 가리키는 배열은 항상 같습니다.

이는 매우 중요합니다. 리액트가 언제 상태 변수를 업데이트 해야할지를 알려주기 때문입니다.
`useEffect`나 `useMemo`/`useCallback` hook을 배우면서 이것이 어떤 의미를 갖는지 살펴보겠습니다.

<details>
<summary>섹션 요약</summary>

- 리액트에서 상태를 업데이트할 때는 기존 객체를 변경하지 않고, setter 함수를 통해 새로운 객체를 생성해야 합니다.
- 새로운 객체는 메모리의 별도 위치에 저장되며, 형태가 같더라도 서로 다른 참조를 가집니다.
- 변경되지 않은 상태 변수는 스냅샷 간에 동일한 참조가 재사용됩니다. 이 특성은 `useEffect`, `useMemo` 등의 동작에 중요한 역할을 합니다.

</details>

### 🎆 Snapshot이 정확히 뭔가요?

저는 "스냅샷" 이라는 용어를 렌더링을 수행한 결과로 사용했습니다. 이는 두가지 요소의 결합입니다.

1. 렌더링이 발생했을 때, props/state의 값
2. 컴포넌트에서 반환된 리액트 요소.

여기서 궁금증이 생길 수 있습니다. *snapshot*과 *instance*의 차이가 뭘까요?

컴포넌트의 인스턴스는 해당 컴포넌트에 대한 정보가 담겨있는("진실의 원천") 자바스크립트 객체입니다. 이 객체는 컴포넌트가 마운트될 때 생성되고 언마운트될 때까지 유지됩니다.

이와 반대로 스냅샷은 자바스크립트 객체가 아닙니다. 좀 더 추상적인 개념입니다. 스냅샷은 특정 시점의 데이터를 의미합니다.

즉, 인스턴스가 상태의 실제 값을 보관하고 있고, 상태가 변경될 때마다 그 시점의 값을 캡처한 스냅샷이 만들어진다고 할 수 있습니다.

좀더 자세한 내용을 확인하고 싶다면 ["State as a Snapshot"](https://react.dev/learn/state-as-a-snapshot)를 확인해보세요.

<details>
<summary>섹션 요약</summary>

- 스냅샷은 렌더링 시점의 props/state 값과 반환된 리액트 요소의 결합으로, 특정 시점의 데이터를 캡처한 추상적인 개념입니다.
- 인스턴스는 컴포넌트의 실제 정보를 담고 있는 자바스크립트 객체로, 마운트부터 언마운트까지 유지됩니다.
- 인스턴스가 상태의 실제 값을 보관하고, 상태가 변경될 때마다 그 시점의 스냅샷이 만들어집니다.

</details>

## 🌃 Refs

일반적으로 웹 개발자는 `div`, `button`, `form` 같은 표준 DOM요소를 이용하여 사용자 인터페이스를 구축합니다.

하지만 웹은 HTML canvas처럼 UI를 그리는 다른 방식을 제공합니다.

HTML Canvas는 익숙하지 않으신 분들을 위해 설명드리자면, 마치 마이크로소프트 페인트처럼 도형을 그려 그래픽을 만들 수 있는 기능을 제공합니다.

HTML canvas로 작업을 시작하려면, `canvas` 태그를 렌더링해야 한후에 자바스크립트 코드를 작성해야 합니다. 아래는 예시입니다.

```javascript
const canvas = document.querySelector('canvas');
const ctx = canvas.getContext('2d');

ctx.fillRect(0, 0, 200, 100);
```

`ctx`는 그림을 그리는 영역입니다. Webgl을 사용하여 2D로 그릴것인지 3D로 그릴것인지 선택할 수 있습니다.

여기서 생기는 의문점은 어떻게 이런 작업을 리액트에서 할 수 있을지 입니다.

JSX를 작성하는것은 사실 HTML을 작성하는게 아닙니다. 자바스크립트를 작성하면 리액트가 코드를 실제 DOM으로 변환합니다.

이런 추상화는 리액트가 어려운 작업을 대신해주기 때문에 좋습니다. 하지만 때때로 이 점이 방해가 될때가 있습니다. 바로 DOM 노드에 접근하려고 할 때 입니다.
이런 경우는 `canvas`를 사용할 때 발생합니다.

앞서 살펴본 코드에서는 `document.querySelector`를 사용하여 렌더링 후에 요소의 참조값을 가져왔지만 이 방법은 리액트에서 권장하는 방법이 아닙니다.

리액트는 이런 상황에서 사용할 수 있는 `ref`를 제공합니다.

```jsx
<canvas
  width={200}
  height={100}
  ref={function (canvas) {
    console.log(canvas);
  }}
/>
```

리액트가 이 요소를 렌더링할 때마다, `ref` 함수가 호출되고 DOM요소를 전달합니다.

만약 canvas의 참조값을 인자로 받아 그림을 그리는 `draw` 함수를 작성했다면 아래와 같이 작성할 수 있습니다.

```jsx
<canvas
  width={200}
  height={200}
  ref={function (canvas) {
    draw(canvas);
  }}
/>
```

이 방식은 잘 동작하지만 UX적으로는 좋지 않습니다. 버튼을 클릭할 때 그림을 그리게끔 수정해보겠습니다.

버튼의 `onClick` 핸들러에서 canvas의 ref에 접근하려면 해당 변수를 저장해두어야 합니다.

```jsx
function ArtGallery() {
  let canvas;

  return (
    <main>
      <div className="canvas-wrapper">
        <canvas
          ref={function (ref) {
            canvas = ref;
          }}
          width={200}
          height={200}
        />
      </div>
      <button
        onClick={() => {
          draw(canvas);
        }}
      >
        Draw!
      </button>
    </main>
  );
}

function draw(canvas) {
  // Code omitted
}
```

이 방법은 놀라울 정도로 잘 동작하지만 성능적인 문제가 있습니다. `ref` 함수는 모든 렌더링마다 실행되기 때문에 계속 DOM 노드를 탐색합니다.

컴포넌트가 최초 렌더링될 때 한번만 조회했으면 좋을것 같습니다. 그런 다음에 canvas의 참조값을 이후 모든 렌더링 과정에 전달하면 됩니다.

이는 `useRef` hook을 사용하면 가능합니다.

```jsx
function ArtGallery() {
  const canvasRef = useRef(); // {current: undefined}

  return (
    <main>
      <div className="canvas-wrapper">
        <canvas
          // 2. Capture a reference to the <canvas> tag,
          // and put it in the “canvasRef” box.
          //
          // { current: <canvas> }
          ref={canvasRef}
          width={200}
          height={200}
        />
      </div>
      <button
        onClick={() => {
          // 3. Pluck the <canvas> tag from the box,
          // and pass it onto our `draw` function.
          draw(canvasRef.current);
        }}
      >
        Draw!
      </button>
    </main>
  );
}
```

`useRef`를 호출하면 `current` 프로퍼티를 가진 객체를 반환합니다.

이 객체를 `ref` 속성에 전달하면 리액트는 이 객체의 `current` 프로퍼티를 canvas의 참조값으로 변경합니다.

컴포넌트가 최초로 렌더링될 때 실행된다는 점은 성능상으로 이득입니다.

이 방법은 DOM 노드의 참조값을 사용할 때 리액트에서 권장하는 방법입니다.

:::info DOM 노드만 해당되지는 않습니다!
`useRef`로 생성된 ref 객체는 일종의 상자로 생각할 수 있습니다.
상자에 원하는 무엇이든 다 저장할 수 있습니다.(DOM 노드, 숫자, 배열, 객체, 함수 등등)
:::

<details>
<summary>섹션 요약</summary>

- 리액트는 DOM을 추상화하기 때문에, DOM 노드에 직접 접근해야 할 때(예: canvas) `ref`를 사용합니다.
- `ref` 콜백 함수 방식은 매 렌더링마다 실행되어 성능 문제가 있을 수 있습니다.
- `useRef` hook은 `{current: ...}` 객체를 반환하며, 최초 렌더링 시 한 번만 DOM 참조를 캡처하고 이후 렌더링에서 재사용합니다.
- ref 객체는 DOM 노드뿐 아니라 어떤 값이든 저장할 수 있는 범용 상자입니다.

</details>

## 🌃 Side Effect

애플리케이션을 만들다 보면, 외부 시스템과 동기화 해야하는 상황이 생깁니다. 예를 들어 아래와 같으 상황입니다.

1. 네트워크 요청 보내기
2. timeout/interval 관리하기
3. Local Storage 읽고/쓰기
4. 전역 이벤트 감지하기

리액트는 이런 것들을 "side effect"라고 부릅니다. 이 섹션에서는 side effect를 효과적으로 사용하는 방법을 알아보겠습니다.

### 🎆 useEffect hook 살펴보기

[리액트의 핵심 동작 흐름](/blog/useState-hook#-리액트의-핵심-동작-흐름)에서 살펴본 것처럼 리액트는 JSX을 바탕으로 DOM을 생성합니다.
JSX가 변경되면(상태나 props의 변경으로) 리액트는 변경사항과 항상 동기화된 상태를 유지하기 위해 DOM을 업데이트 합니다. 이는 리액트의 주된 작업이자, 목적입니다.

만약 리액트의 역할 범위를 넘어가는 작업은 어떨까요? 예를 들어, 상태 변수와 동기화 되도록 document의 제목을 변경하려는 작업같은 거요.

이런 작업을 **side effect**라고 합니다. Side effect는 local storage와 상호작용하거나, 네트워크 요청을 보내는 작업을 포함합니다.

리액트는 이런 side effect를 효율적으로 처리하기위해 `useEffect`라는 hook을 제공합니다. `useEffect`에 콜백함수로 전달한 코드는 렌더링 후에 호출됩니다.

컴포넌트 내부에서 그냥 코드를 작성하면 안될까요? 가능은 합니다. 하지만 `useEffect`는 추가적인 도구를 제공합니다.
의존성을 전달해 `useEffect`가 실행될 조건을 추가할 수 있습니다. 전달한 의존성이 변경된다면 `useEffect`의 콜백 함수가 실행됩니다.

```jsx
React.useEffect(() => {
  // Effect logic
  document.title = `Count: ${count}`;
}, [count]);
```

`[count]`를 전달함으로써, effect 로직이 `count`가 변경될 때마다 실행된다고 알려줍니다.

`useEffect`는 전달한 의존성이 무엇이든 간에 최초 렌더링시에 항상 실행됩니다.

### 🎆 Strict Mode

`useEffect`에 대해 자세히 알아보기 전에 Strict Mode에 대해 간략히 짚고 넘어가겠습니다.

리액트 프로젝트를 진행할 때, 뭔가 이상한 점을 발견할 수 있습니다. 그것은 컴포넌트의 최초 마운트시에 effect가 두번 실행된다는 것입니다.

이는 _Strict Mode_ 때문입니다. Strict Mode에서는 개발 환경에서 특정 코드를 재실행합니다. 이는 잠재적인 문제를 찾기위함 입니다.
자세한 내용은 뒤에서 살펴보겠습니다.

<details>
<summary>섹션 요약</summary>

- Side effect는 네트워크 요청, timeout/interval 관리, Local Storage, 전역 이벤트 감지 등 외부 시스템과의 동기화 작업을 의미한다
- `useEffect`는 렌더링 후에 side effect를 실행하며, 의존성 배열을 통해 실행 조건을 제어할 수 있다
- `useEffect`는 의존성과 관계없이 최초 렌더링 시 항상 실행된다
- Strict Mode에서는 개발 환경에서 잠재적 문제를 찾기 위해 effect가 두 번 실행된다

</details>

## 🌃 useEffect의 Lint 규칙

아래와 같은 코드가 있다고 가정해보겠습니다.

```jsx
function App() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    console.log(count);
  }, []);

  return (
    <>
      <p>The count is: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </>
  );
}
```

위 코드를 실행하려고 하면, `react-hooks/exhaustive-deps`라는 규칙을 위반했다는 lint 오류가 노출됩니다.

이는 흔히 저지르는 실수이며, 이 문제를 해결하기 위한 글들이 꽤 많습니다. 하지만, 가장 흔한 해결책은 문제를 수정하는것이 아니라 규칙을 비활성화 하는 것 입니다.

```
useEffect(() => {
  console.log(count);
  // eslint-disable-next-line
}, []);
```

이 주석은 경고를 끄는것이지 해결하는것은 아닙니다.

이 규칙은 이유없이 추가된것이 아닙니다. 만약 `count` 상태값을 의존성 배열에 추가하지 않으면 혼란스러운 상황을 마주할 수 있습니다.

`setCount`를 호출하여 재렌더링이 될 때마다 새로운 스냅샷이 생성되고 새로운 `count` 변수가 생성됩니다. 이는 매 스냅샷마다 새로운 `count` 변수값이 존재한다는 의미입니다.

`count` 변수를 의존성 배열에 추가하지 않으면, 해당 `effect`는 특정 시점에 고정됩니다. `console.log(count)` 코드는 최초의 `count` 값인 `0`을 가지고 있습니다.

이는 상태가 변경될 때마다 당황스러운 상황을 마주할 수 있습니다.

```jsx
function App() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    // This 'count' is equal to 0
    console.log(count);
    // eslint-disable-next-line
  }, []);

  return (
    <>
      {/* this `count` is equal to 3: */}
      <p>The count is: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </>
  );
}
```

Effect 코드는 상태를 참조하는 다른 코드와 같은 상태로 동기화되지 않습니다. 즉, 같은 시점에 여러 버전의 상태가 존재하는 상황입니다.

이를 수정하기 위해서는 `count`를 의존성 배열에 추가해야 합니다.

```jsx
useEffect(() => {
  console.log(count);
}, [count]);
```

위 코드의 의미는 `setCount`가 호출될 때마다 effect가 다시 실행된다는 의미입니다.

<details>
<summary>섹션 요약</summary>

- `react-hooks/exhaustive-deps` lint 규칙은 effect 내에서 사용하는 값을 의존성 배열에 포함하도록 강제한다
- `eslint-disable-next-line`으로 규칙을 비활성화하는 것은 경고를 끄는 것이지 문제를 해결하는 것이 아니다
- 의존성 배열에 참조하는 상태를 누락하면 effect가 특정 시점의 스냅샷에 고정되어 최신 상태와 동기화되지 않는다
- 해결 방법은 effect에서 사용하는 상태 변수를 의존성 배열에 추가하는 것이다

</details>

## 🌃 마운트될 때 실행하기

컴포넌트가 마운트될 때 자동으로 포커싱이 되는 텍스트 인풋을 만들어보겠습니다.

```jsx
function App() {
  const [searchTerm, setSearchTerm] = useState('');

  return (
    <main>
      <form>
        {/* 마운트될 때 인풋 포커싱 */}
        <input
          value={searchTerm}
          onChange={(e) => setSearchTerm(e.target.value)}
        />
        <button>Search</button>
      </form>
    </main>
  );
}
```

앞서 살펴본 `useRef` hook을 사용하여 인풋 요소을 참조해야 합니다.

```jsx
function App() {
  const [searchTerm, setSearchTerm] = useState('');

  const inputRef = useRef();

  return (
    <main>
      <form>
        <input
          ref={inputRef}
          value={searchTerm}
          onChange={(e) => setSearchTerm(e.target.value)}
        />
        <button>Search</button>
      </form>
    </main>
  );
}
```

인풋 DOM은 `.focus()` 메서드를 가지고 있어 포커싱을 위해 호출할 수 있습니다. 하지만 어떻게 마운트될 때 호출할 수 있을까요?

우선 렌더링 과정에서 시도해보겠습니다.

```jsx
function App() {
  const [searchTerm, setSearchTerm] = useState('');

  const inputRef = useRef();
  inputRef.current.focus();
}
```

안타깝게도 이 코드는 오류가 발생합니다.

:::danger
Cannot read properties of undefined (reading 'focus')
:::

문제는 `inputRef`가 생성될 때 빈값으로 생성된다는 것입니다(`{current: undefined}`). 최초의 렌더링 후에 DOM 노드에 접근할 수 있습니다.

이를 해결하기 위해서 `useEffect` hook을 사용해야 합니다.

```jsx
function App() {
  const [searchTerm, setSearchTerm] = useState('');

  const inputRef = useRef();

  useEffect(() => {
    inputRef.current.focus();
  }, []);
}
```

반드시 빈 배열을 의존성 배열에 전달해야 합니다. 이는 해당 effect가 어떠한 값에도 의존하지 않으므로 다시 실행되지 않습니다.

아래의 코드는 완성된 코드입니다.

```jsx live
function App() {
  const [searchTerm, setSearchTerm] = React.useState('');

  const inputRef = React.useRef();

  React.useEffect(() => {
    inputRef.current.focus();
  }, []);

  return (
    <>
      <main>
        <form>
          <input
            ref={inputRef}
            value={searchTerm}
            onChange={(event) => {
              setSearchTerm(event.target.value);
            }}
          />
          <button>Search</button>
        </form>
      </main>
    </>
  );
}
```

:::warning "autofocus" 속성
HTML form 인풋은 페이지가 로드될 때 자동으로 포커싱을 받게되는 `autofocus` 속성을 가지고 있습니다.

```html
<input autofocus type="text" />
```

인풋에 자동으로 포커스를 맞추는 내장 기능이 있는데 왜 `useRef`나 `useEffect`를 사용할 필요가 있을까요?

아쉽게도, 리액트에서 `autofocus` 속성을 사용하는것은 안전하지 않습니다.

`autofocus` 속성은 페이지 로딩시점에 요소가 제공될 때만 안정적으로 동작합니다. 만약 요소가 나중에 페이지에 동적으로 삽입되는 경우라면 동작하지 않을 것입니다.

리액트에서는 모든 요소가 동적으로 삽입됩니다. 예외가 있다면 서버 사이드 렌더링을 사용하는 경우인데, 이 경우에도 사용자가 사이트에 처음 방문하는 페이지에만 해당됩니다.

위에서 살펴본 코드에서, ref로 인풋 요소를 참조하여 `.focus()` 메서드를 effect 안에서 실행하는것은 가장 정석적인 방법입니다.
:::

### 🎆 구독하기

사용자의 마우스 커서 위치를 추적해야할 필요가 있다고 가정해보겠습니다. 마우스를 움직일 때마다 상태를 업데이트 해야합니다.

아래 코드처럼 `onMouseMove` 이벤트 핸들러를 DOM 노드에 추가할 수 있습니다.

```jsx
<div
  onMouseMove={(e) => {
    setMousePosition({
      x: e.clientX,
      y: e.clientY,
    });
  }}
></div>
```

이 코드는 `<div>` 요소에 호버될 때만 실행됩니다. 만약 마우스가 뷰포트 내 어디에 있든 상관없이 커서 위치를 추적하고 싶다면 어떻게 해야 할까요?

전역 마우스 이벤트를 구독하기 위해서, `window.addEventListener`를 사용해야 합니다. 전역 이벤트를 effect hook안에서 아래와 같이 구독할 수 있습니다.

```jsx
useEffect(() => {
  function handleMouseMove(e) {
    setMousePosition({
      x: e.clientX,
      y: e.clientY,
    });
  }

  window.addEventListener('mousemove', handleMouseMove);
});
```

얼핏 봤을 때 잘 동작하는거 같습니다. 하지만 이 방법에는 두가지 큰 문제가 있습니다.

첫번째로 이벤트 리스너를 제거하지 않습니다. 이에 대해서는 뒤에서 살펴보겠습니다.

이번 섹션에서는 여러개의 이벤트가 추가되는 문제에 대해 살펴보겠습니다.

의존성 배열을 추가하지 않았기 때문에 모든 렌더링마다 effect가 실행됩니다. 즉, 사용자의 마우스 위치가 변경될 때마다 `window.addEventListener`가 호출됩니다.
만약 마우스를 100번 움직이면 이벤트 리스너가 100번 추가됩니다.

`window.addEventListener`는 구독을 하는 방식입니다. 맨 처음 마운트될 때 한번만 구독하면 됩니다.

아래는 위의 문제를 해결한 코드입니다.

```jsx
useEffect(() => {
  function handleMouseMove(e) {
    setMousePosition({
      x: e.clientX,
      y: e.clientY,
    });
  }

  window.addEventListener('mousemove', handleMouseMove);
}, []);
```

`window.addEventListener`는 리액트에서 제공하는 함수가 아니라 DOM에서 제공하는 함수입니다.
해당 메소드를 호출하면 `mousemove` 이벤트가 발생할 때마다 함수가 호출되는 장기 실행 프로세스를 설정할 수 있습니다.

아래의 상황들은 위 예시와 비슷한 상황입니다.

- 인터벌 함수 실행
- 웹 소켓 연결
- `ResizeObserver` 사용

아래는 위 코드에서 어떤일이 일어나는지를 설명해주는 그림입니다.

의존성 배열이 비어있을 때는 아래와 같이 동작합니다.

<EffectSubscriptionFirstRender width="100%" height="auto" />

아무런 의존성을 추가하지 않으면 아래와 같이 동작합니다.

<EffectSubscriptionEveryRender width="100%" height="auto" />

:::warning Clean up 함수의 부재
위 코드에서 이벤트 리스너를 추가했지만 제거하지는 않았습니다. 이 부분도 문제입니다.
이렇게 될 경우 메모리 누수를 야기할 수 있기때문입니다.

이후에 clean up 하는 방법을 살펴보겠습니다.
:::

<details>
<summary>섹션 요약</summary>

- 렌더링 과정에서는 ref가 아직 연결되지 않아 DOM에 접근할 수 없으므로, 마운트 시 DOM 조작은 `useEffect`에 빈 의존성 배열(`[]`)을 전달하여 처리한다
- `autofocus` 속성은 리액트에서 동적으로 삽입되는 요소에 안정적으로 동작하지 않으므로, ref와 `useEffect`를 사용하는 것이 정석이다
- `window.addEventListener` 같은 전역 이벤트 구독은 빈 의존성 배열로 마운트 시 한 번만 실행해야 하며, 의존성 배열을 생략하면 매 렌더링마다 중복 구독된다

</details>

## 🌃 Clean up 하기

앞서 `useEffect` hook을 활용하여 마운트될 때 오래 실행되는 작업을 추가하는 방법을 살펴보았습니다. 하지만 이 방법은 불완전합니다.

문제를 살펴보기 위해 앞에서 살펴본 코드를 조건부로 렌더링하기 위해 변경해보겠습니다.

```jsx
function App() {
  const [isTrackingMouse, setIsTrackingMouse] = useState(true);

  function toggleMouseTracking() {
    setIsTrackingMouse(!isTrackingMouse);
  }

  return (
    <div className="wrapper">
      <button onClick={toggleMouseTracking}>Toggle Mouse Tracking</button>
      {isTrackingMouse && <MouseTracker />}
    </div>
  );
}
```

`isTrackingMouse`가 truthy한 값일 때, 컴포넌트가 마운트 됩니다. 앞서 살펴봤듯이 컴포넌트가 마운트되면 컴포넌트의 데이터들이 저장되어있는 인스턴스가 생서됩니다.

하지만 `isTrackingMouse`가 falsy한 값이 되면 컴포넌트가 언마운트되며 인스턴스와 관련된 모든 DOM이 제거됩니다.

따라서 컴포넌트가 언마운트될 때 그와 연관된 모든 effect들도 제거된다고 생각할 수도 있습니다. 하지만 실제로 그렇지 않습니다.

`MouseTracker` 컴포넌트는 아래의 effect를 가지고 있습니다.

```jsx
useEffect(() => {
  function handleMouseMove(e) {
    setMousePosition({
      x: e.clientX,
      y: e.clientY,
    });
  }

  window.addEventListener('mousemove', handleMouseMove);
}, []);
```

컴포넌트가 언마운트될 때, 이벤트 리스너는 제거되지 않고 계속해서 커서의 포지션을 추적하며 존재하지 않는 컴포넌트 인스턴스의 `setMousePosition`을 호출합니다.

이것이 문제가 되는 이유는 크게 두가지 입니다.

1. 컴포넌트가 다시 마운트될 때마다, 이전의 리스너가 제거되지 않은채로 새로운 이벤트 리스너가 추가됩니다.
2. `setMousePosition` 함수를 호출함으로써 이전 컴포넌트 인스턴스를 참조하기 때문에, 자바스크립트 가비지컬렉터는 이전 인스턴스를 제거하지 않습니다.
   즉, 해당 컴포넌트가 마운트될 때마다 절대 지워지지 않은 인스턴스를 생성하는 샘입니다.

이는 메모리 누수를 야기합니다. 애플리케이션을 오래 사용하면 할 수록 더 많은 메모리를 소비하게 됩니다.
새로고침을 하게되면 메모리가 해제되지만 대부분의 사용자는 같은 탭을 오래 열어둡니다.

그렇다면 왜 이벤트 리스너는 자동으로 제거되지 않을까요? 리액트는 사실 effect 함수에서 어떤일이 일어나는지 모르기 때문입니다.

```jsx
// 리액트가 실제로 보는것
useEffect(() => {
  // ???
}, []);
```

리액트는 우리가 함수를 전달했다는 것과, 언제 해당 함수를 실행해야 하는지(의존성 배열을 통해)만 알고 함수의 내부는 모릅니다.
따라서 리액트는 이벤트 리스너를 구독했다는것을 모릅니다.

마찬가지로 이벤트 리스너를 실행하는 자바스크립트 엔진은 이벤트 리스너가 컴포넌트 인스턴스 내부에 생성되어 컴포넌트의 존재에 따라 다르게 동작해야한다는 것을 알지 못합니다.

즉, 두개의 독립된 시스템(리액트와 DOM의 이벤트 리스너)이 존재하고 있으며, 이 두개를 동기화하는것은 사용하는 사람의 역할입니다.

다행히도, `useEffect`는 **cleanup** 함수를 제공함으로써 이를 가능하게 합니다.

아래은 cleanup 함수를 적용한 예시입니다.

```jsx
useEffect(() => {
  function handleMouseMove(e) {
    setMousePosition({
      x: e.clientX,
      y: e.clientY,
    });
  }

  window.addEventListener('mousemove', handleMouseMove);

  return () => {
    window.removeEventListener('mousemove', handleMouseMove);
  };
}, []);
```

Effect 함수 안에서, cleanup 작업을 포함하는 함수를 리턴하고 있습니다. 리액트는 이 함수를 저장하고 있다가 적절한 시점에 실행시킵니다.
그 시점은 컴포넌트가 언마운트되기 직전입니다.

이는 전형적인 구독 패턴입니다. Effect 함수에서 구독을 하고 cleanup 함수에서 제거합니다.
리액트는 cleanup 함수를 컴포넌트 언마운트 직전에 실행하여 장기적으로 실행되는 작업을 멈추고 메모리 누수를 방지할 수 있도록 도와줍니다.

:::info 함수를 반환하는 함수
커리어 초반에 가장 헷갈렸던것들 중 하나는 언제 함수가 함수를 반환하는지 였습니다. 아래와 같은 코드들이 어려웠습니다.

```js
function doSomething() {
  const hi = 5;

  return function () {
    return hi * 2;
  };
}
```

`useEffect`의 cleanup API가 위와 비슷하게 동작합니다. Effect 함수는 함수를 반환합니다.

```js
useEffect(() => {
  // Effect logic
  return () => {
    // Cleanup logic
  };
});
```

좋은 소식은 구현 방식까지 깊게 살펴볼 필요는 없다는 것입니다.

가장 중요한 것은, 핵심 아이디어를 이해하는 것입니다. 리액트에게 effect 함수와 cleanup 함수 두개를 전달하면, 리액트는 적절한 시점에 두개의 함수를 호출합니다.
:::

### 🎆 Cleanup 함수와 의존성

앞에서 장기적인 작업이 컴포넌트의 라이프 사이클과 동기화되면서 effect에 추가되고 제거되는 방식을 살펴보았습니다.
하지만 effect에 의존성이 있으면 어떻게 될까요? 이 의존성들은 cleanup 함수와 어떻게 상호작용할까요?

부모에서 `MouseTracker` 컴포넌트를 마운트/언마운트하면서 제어하는것 대신에, `MouseTracker` 컴포넌트 내부에서 제어하도록 코드를 변경해보겠습니다.

```jsx
function MouseTracker() {
  const [mousePosition, setMousePosition] = useState({ x: 0, y: 0 });
  const [isEnabled, setIsEnabled] = useState(true);

  useEffect(() => {
    function handleMouseMove(e) {
      setMousePosition({
        x: e.clientX,
        y: e.clientY,
      });
    }

    window.addEventListener('mousemove', handleMouseMove);

    return () => {
      window.removeEventListener('mousemove', handleMouseMove);
    };
  }, []);

  function toggleMouseTracking() {
    setIsEnabled(!isEnabled);
  }

  return (
    <>
      <button onClick={toggleMouseTracking}>Toggle Mouse Tracking</button>
      <p>
        {mousePosition.x} / {mousePosition.y}
      </p>
    </>
  );
}
```

`isEnable` 값이 `true`일 때만 이벤트 리스너를 등록하고 싶습니다. 그리고 `isEnable` 값이 `false`일 때 이벤트 리스너를 해제하려고 합니다.

처음 effect를 사용했을 때, 아래와 같이 생각했었습니다.

```js
if (isEnabled) {
  useEffect(() => {
    function handleMouseMove(e) {
      setMousePosition({
        x: e.clientX,
        y: e.clientY,
      });
    }

    window.addEventListener('mousemove', handleMouseMove);

    return () => {
      window.removeEventListener('mousemove', handleMouseMove);
    };
  }, []);
}
```

조건부로 `useEffect`를 호출하는게 얼핏보면 합당해보입니다. 하지만 앞에서 살펴보았듯이 위 코드는 hook의 규칙을 위반했습니다. 어떤 hook이든 조건부로 호출하면 안됩니다.

좋습니다. 그렇다면 `if` 조건문을 hook 내부로 이동해보겠습니다.

```jsx
useEffect(() => {
  if (isEnabled) {
    function handleMouseMove(e) {
      setMousePosition({
        x: e.clientX,
        y: e.clientY,
      });
    }

    window.addEventListener('mousemove', handleMouseMove);

    return () => {
      window.removeEventListener('mousemove', handleMouseMove);
    };
  }
}, []);
```

위 코드를 실행하면 다른 lint 경고가 노출됩니다. 의존성을 누락했다구요! [useEffect의 Lint 규칙](#-useeffect의-lint-규칙)에서 살펴보았듯이 모든 상태 변수를 의존성 배열에 추가해야 합니다.

`isEnabled`를 의존성 배열에 추가하면 lint 오류가 해결됩니다.

```js
useEffect(() => {
  if (isEnabled) {
    function handleMouseMove(e) {
      setMousePosition({
        x: e.clientX,
        y: e.clientY,
      });
    }

    return () => {
      window.removeEventListener('mousemove', handleMouseMove);
    };
  }
}, [isEnabled]);
```

위 코드는 어떻게 동작하는걸까요?

1. 컴포넌트가 마운트되면, effect가 실행되고 이벤트 리스너가 등록됩니다. 또한 cleanup 함수가 리액트에게 전달됩니다.
2. 사용자가 마우스를 움직이면 `mousePosition` 상태가 빠르게 업데이트 되지만, effect는 재실행되지 않습니다. `mousePosition`이 의존성 배열에 추가되지 않았기 때문입니다.
3. 사용자가 버튼을 클릭하면 `isEnabled`는 false로 변경됩니다. `isEnabled`가 의존성 배열에 있기때문에 effect는 재실행됩니다.
4. 하지만 그전에 cleanup 함수가 먼저 실행됩니다.
5. 이벤트 리스너는 제거되고 `isEnabled` 값이 `false`가 되기때문에 effect의 이벤트 리스너는 더이상 등록되지 않습니다.
6. 위 과정은 사용자가 버튼을 클릭할 때마다 반복됩니다.

- 컴포넌트가 언마운트되어도 effect에서 등록한 이벤트 리스너는 자동으로 제거되지 않아 메모리 누수가 발생한다
- `useEffect`의 콜백에서 cleanup 함수를 반환하면, 리액트가 컴포넌트 언마운트 직전에 이를 실행하여 구독을 해제한다
- 의존성이 변경되어 effect가 재실행될 때도 이전 cleanup 함수가 먼저 실행된 후 새로운 effect가 실행된다
- Hook은 조건부로 호출할 수 없으므로, 조건 분기는 `useEffect` 내부에 두고 해당 조건 변수를 의존성 배열에 추가해야 한다

## 🌃 Stale Values

이 섹션에서는 구현된 미디어 플레이어에 스페이스바를 누를때 현재 재생중인 음악이 멈추는 기능을 추가해보며 stale한 값들에대해 살펴보겠습니다.

```jsx
import React from 'react';
import { Play, Pause } from 'react-feather';

import VisuallyHidden from './VisuallyHidden';

function MediaPlayer({ src }) {
  const [isPlaying, setIsPlaying] = React.useState(false);

  const audioRef = React.useRef();

  return (
    <div className="wrapper">
      <div className="media-player">
        <img
          alt=""
          src="https://sandpack-bundler.vercel.app/img/take-it-easy.png"
        />
        <div className="summary">
          <h2>Take It Easy</h2>
          <p>Bvrnout ft. Mia Vaile</p>
        </div>
        <button
          onClick={() => {
            if (isPlaying) {
              audioRef.current.pause();
            } else {
              audioRef.current.play();
            }

            setIsPlaying(!isPlaying);
          }}
        >
          {isPlaying ? <Pause /> : <Play />}
          <VisuallyHidden>Toggle playing</VisuallyHidden>
        </button>

        <audio
          ref={audioRef}
          src={src}
          onEnded={() => {
            setIsPlaying(false);
          }}
        />
      </div>
    </div>
  );
}
```

먼저 관련된 이벤트를 추가하겠습니다.

```jsx
useEffect(() => {
  function handleKeyDown(event) {
    if (event.code === 'Space') {
      // TODO: Play of pause
    }
  }

  window.addEventListener('keydown', handleKeyDown);

  return () => {
    window.removeEventListener('keydown', handleKeyDown);
  };
}, []);
```

`keydown` 이벤트를 구독하여 스페이스바 누름 여부를 판단합니다. Cleanup 함수는 이 이벤트들 구독 해제합니다.
빈 배열을 의존성에 추가했기 때문에 이벤트 리스너는 컴포넌트의 라이프사이클에서 딱 한번 실행됩니다.

이제 음악을 어떻게 재생/중지 할까요? 이미 클릭했을 때 재생/중지 하는 코드가 있습니다.

```jsx
<button
  onClick={() => {
    if (isPlaying) {
      audioRef.current.pause();
    } else {
      audioRef.current.play();
    }

    setIsPlaying(!isPlaying);
  }}
/>
```

위 코드에서 관리해야할 상태가 두개라는 점이 문제를 조금 까다롭게 만듭니다.

1. `isPlaying` 상태 변수
2. 실제 노래의 재생 여부를 판단하는 오디오 DOM의 내부 상태

한가지 옵션은 두개를 한꺼번에 제어하는 함수를 생성하는 것 입니다.

```js
function togglePlaying() {
  if (isPlaying) {
    audioRef.current.pause();
  } else {
    audioRef.current.play();
  }

  setIsPlaying(!isPlaying);
}
```

위와 같이 함수를 만든 다음 버튼을 클릭할 때나 `handleKeyDown` 메소드 내부에서 호출하면 됩니다.

하지만 이 방법도 두가지 문제가 있습니다.

1. `togglePlaying` 함수를 effect의 의존성 배열에 추가해야하는데, 이렇게 하면 성능 문제가 발생할 수 있습니다.
2. 정의된 함수를 사용하는것을 잊기 쉽습니다.

두 번째 문제를 좀 더 자세히 살펴보면, 다른 개발자나 함수를 만든 본인도 관련된 상태가 두 개라는 사실을 깨닫기 어려울거 같습니다.

우리가 주의를 기울이지 않는다면, 두 상태 변수가 분리되어 업데이트 되는 상황에 처할 수 있습니다.
`isPlaying` 상태가 true로 바뀌어도 오디오가 재생되지 않을 수 있습니다. 이 문제는 디버깅하기 어렵습니다.

두번째 `useEffect` hook을 추가하여 해결할 수 있습니다. 이 hook은 오디오 DOM의 내부 상태를 `isPlaying` 상태 변수와 동기화해 줍니다.

```js
useEffect(() => {
  if (isPlaying) {
    audioRef.current.play();
  } else {
    audioRef.current.pause();
  }
}, [isPlaying]);
```

이렇게 하면 버튼은 상태 변수만 토글하면 됩니다.

```jsx
<button
  onClick={() => {
    // This work is no longer necessary:
    //
    //   if (isPlaying) {
    //     audioRef.current.pause();
    //   } else {
    //     audioRef.current.play();
    //   }

    setIsPlaying(!isPlaying);
  }}
/>
```

그리고 `isPlaying` 상태를 `keydown` 이벤트 내부에서도 업데이트 하면됩니다.

```js
useEffect(() => {
  function handleKeyDown(event) {
    if (event.code === 'Space') {
      setIsPlaying(!isPlaying);
    }
  }

  window.addEventListener('keydown', handleKeyDown);

  return () => {
    window.removeEventListener('keydown', handleKeyDown);
  };
}, []);
```

하지만 이렇게 업데이트 하면 아래와 같은 lint 경고가 노출됩니다.

:::danger
React Hook React.useEffect has a missing dependency: "isPlaying"
:::

문제는 이 effect 코드가 첫번째 렌더링 직후에만 실행되어야 한다는 점입니다. 즉 `isPlaying`의 최초값인 `false`일때의 스냅샷에만 접근할 수 있다는것을 의미합니다.

사용자가 스페이스바를 누를 때 `setIsPlaying(!false)`를 호출하는것과 마찬가지 입니다. 이는 노래를 재생할수는 있어도 정지할 수는 없다는 뜻입니다.

이 부분을 그림으로 표현하면 아래와 같습니다.

이 문제를 해결하기 위한 옵션은 두가지가 있습니다.

1. 의존성 추가하기
2. 콜백함수 사용하기

첫번째 옵션은 앞에서 살펴보았습니다. `isPlaying`을 의존성 배열에 추가하면 됩니다.

```js
useEffect(() => {
  function handleKeyDown(event) {
    if (event.code === 'Space') {
      setIsPlaying(!isPlaying);
    }
  }

  window.addEventListener('keydown', handleKeyDown);

  return () => {
    window.removeEventListener('keydown', handleKeyDown);
  };
}, [isPlaying]);
```

이 방법에서는 `isPlaying` 함수가 변경될 때마다 effect가 재실행되고 cleanup 함수가 실행됩니다. 다시 말해, `keydown` 이벤트의 콜백함수를 계속 새롭게 생성해 최신 스냅샷에 접근하게 합니다.

이런 상황에서 이는 문제가 되지 않습니다. 이벤트 리스너를 추가하고 제거하는것은 매우 빠르기 때문입니다.

아래는 위 과정을 그림으로 나타낸 것입니다.

하지만 만약 구독/구독해제 작업이 느리면 어떻게 될까요? 의존성 배열에 값을 추가하지 않고 `isPlaying`의 최신 값에 접근할 수 있는 방법이 있을까요?

콜백 함수를 쓰면 해결됩니다.

```js
useEffect(() => {
  function handleKeyDown(event) {
    if (event.code === 'Space') {
      setIsPlaying((currentIsPlaying) => {
        return !currentIsPlaying;
      });
    }
  }

  window.addEventListener('keydown', handleKeyDown);

  return () => {
    window.removeEventListener('keydown', handleKeyDown);
  };
}, []);
```

상태 업데이트 함수에 함수를 전달하면, 리액트는 해당 함수를 호출하고 반환하는 값을 새로운 상태로 업데이트 합니다.

예를 들어, 아래의 두 구문은 정확히 동일한 결과를 생성합니다.

```js
setIsPlaying(5);
setIsPlaying(() => {
  return 5;
});
```

리액트가 이 함수를 실행할 때 현재 상태값을 인자로 받습니다. 이 값은 최신 상태의 값이고 컴포넌트 인스턴스로 부터 받아옵니다.

아래는 콜백함수를 썼을 때 동작하는 과정을 그림으로 나타낸 것입니다.

이 접근 방식은 괜찮을까요? 두가지 방법 모두 대부분의 경우 동일하게 동작합니다.
이 방식이 특정 상황에서 성능적인 이점이 있지만 그 정도는 미미합니다.

:::info Event bubbling
위에서 구현한 코드는 UX적인 측면에서 버그가 있습니다.
버튼에 포커스를 두고 스페이스를 누르면 노래가 재생되자마자 정지됩니다.

이는 브라우저에 내장된 접근성 때문입니다. 버튼에 포커스를 두고 스페이스나 엔터를 누르면 클릭이벤트가 실행됩니다.

버튼의 `onClick` 이벤트와 전역 `keydown` 이벤트가 동시에 실행되기 때문에 이것이 문제가 됩니다. 따라서 `isPlaying` 상태가 빠르게 전환됩니다.

이 문제를 해결하기 위해서는 버튼에 아래의 코드를 추가해야 합니다.

```jsx
<button
  onKeyDown={(event) => {
    if (event.code === 'Space') {
      event.stopPropagation();
    }
  }}
  onClick={() => {
    setIsPlaying(!isPlaying);
  }}
/>
```

위 코드의 동작을 이해하려면 *event bubbling*에 대해 알아야 합니다.

DOM 노드에서 이벤트를 트리거하면 이벤트는 트리의 상위로 전파됩니다.
Keydown 이벤트가 버튼에서 트리거 되면 부모와 조상 노드에도 트리거 됩니다.

이것이 전역 이벤트를 등록할 수 있는 이유입니다! `window` 객체는 DOM에서 최상위 요소이기 때문에 모든 이벤트들이 해당 객체로 전파됩니다.

보통, 버튼에 포커스를 두고 "스페이스"를 눌러 이벤트를 트리거하면 해당 이벤트가 버튼뿐만 아니라 `window` 객체와 그 사이의 모든 요소에서도 발생합니다.
하지만 `event.stopPropagation()`을 호출하면 이 전파를 막을 수 있습니다.

이 함수는 keydown 이벤트가 상위로 전파되지 않고 버튼 요소에서 멈추게끔 합니다. 이 방법을 사용하면 전역 `keydown` 이벤트는 감지되지 않습니다.
:::

<summary>섹션 요약</summary>

- 빈 의존성 배열을 사용한 effect는 최초 렌더링의 스냅샷에 고정되어, 이후 변경된 상태값에 접근할 수 없는 stale value 문제가 발생한다
- 관련된 두 상태(예: `isPlaying`과 오디오 DOM)를 별도의 `useEffect`로 동기화하면, 상태 변수만 업데이트해도 부수 효과가 자동으로 따라온다
- Stale value 해결 방법 1: 의존성 배열에 상태를 추가하여 effect를 재실행한다 (구독/해제가 빠른 경우 적합)
- Stale value 해결 방법 2: 상태 업데이트 함수에 콜백을 전달하면 의존성 추가 없이 최신 상태값에 접근할 수 있다
